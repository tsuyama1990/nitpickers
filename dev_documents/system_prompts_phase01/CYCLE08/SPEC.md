# Cycle 08: Refinement, Dependency Cleanup, and System Stabilization

## 1. Summary
Cycle 08 represents the final polishing and stabilization phase, transforming the raw, optimized codebase into a production-ready, professionally documented repository. This cycle executes primarily during the `finalize-session` command. The first major task is comprehensive documentation generation. The AI will parse the final `SYSTEM_ARCHITECTURE.md`, all individual cycle `SPEC.md` files, and the verified `USER_TEST_SCENARIO.md` to generate a cohesive, accurate, and visually appealing `README.md` and detailed Marimo tutorial notebooks. The second major task is a strict dependency audit. During iterative development, numerous packages might have been added to `pyproject.toml` that are no longer necessary for the final build. A dedicated audit node will cross-reference the actual imports in the codebase against the `pyproject.toml` and autonomously remove unused dependencies, ensuring a minimal and secure attack surface. Finally, an end-to-end integration test is performed, resulting in an automated Pull Request generation to the main branch, signifying project completion.

## 2. System Architecture
This cycle modifies the `finalize_session` workflow within `ac_cdd_core/services/workflow.py` and adds new specialized nodes for documentation generation and dependency auditing.

### File Structure Modification
```ascii
.
├── src/
│   └── ac_cdd_core/
│       ├── services/
│       │   └── **workflow.py**
│       ├── **graph_nodes.py**
│       ├── **templates/README_GENERATION_INSTRUCTION.md**
│       └── **templates/DEPENDENCY_AUDIT_INSTRUCTION.md**
├── pyproject.toml
└── README.md
```

## 3. Design Architecture
The implementation relies on specialized, single-purpose AI nodes and deterministic string parsing.

1.  **Documentation Node**: Create a node that uses `README_GENERATION_INSTRUCTION.md`. It ingests all architectural markdown files and outputs a formatted `README.md` containing an elevator pitch, Mermaid diagrams, installation instructions, and usage guides derived directly from the actual implementation.
2.  **Dependency Audit Node**: Create a node that reads `pyproject.toml` and scans all `.py` files in `src/` for `import` statements. It uses `DEPENDENCY_AUDIT_INSTRUCTION.md` to map standard library imports vs. external packages, identifying dependencies listed in `pyproject.toml` that are never imported. It then safely modifies and rewrites the `pyproject.toml` file.
3.  **PR Generation Logic**: The final step utilizes the `git_ops.py` (assuming it exists from AC-CDD) to create a Pull Request on GitHub. The PR body is generated by the AI, summarizing the total work done across all cycles.

## 4. Implementation Approach
The implementation focuses on the `finalize_session` command execution flow.

1.  **Define Instructions**: Create `README_GENERATION_INSTRUCTION.md` and `DEPENDENCY_AUDIT_INSTRUCTION.md`.
2.  **Implement Docs Generation**: In `src/graph_nodes.py`, implement `generate_docs_node`. This node reads the context, generates the README, and writes it to the root directory. It also generates the single Marimo file defined in `USER_TEST_SCENARIO.md`.
3.  **Implement Dependency Audit**: In `src/graph_nodes.py`, implement `audit_dependencies_node`. This node executes a script (or utilizes a tool like `pip-audit` or `deptry` via subprocess) to identify unused packages and updates the `pyproject.toml` accordingly. It then runs `uv sync` to ensure the environment resolves correctly.
4.  **Modify Finalize Workflow**: In `src/services/workflow.py`, update `finalize_session`. Sequence the execution: `audit_dependencies_node` -> `generate_docs_node` -> execute final test suite in sandbox -> generate Pull Request.

## 5. Test Strategy
Testing this final phase involves verifying document generation and safe file modification.

**Unit Testing Approach**: We will unit test the `audit_dependencies_node` by providing a mocked `pyproject.toml` containing an unused dependency (e.g., `requests`) and a mock source directory that only imports standard libraries. We will assert that the node correctly parses the TOML, removes the unused dependency, and writes valid TOML back to disk.

**Integration Testing Approach**: We will simulate the `finalize-session` command in a temporary repository containing a completed mock project. We will verify that the `README.md` is successfully generated and contains the specific text markers required by the instruction template. We will assert that the `pyproject.toml` is cleaned up. Finally, we will mock the GitHub API client to verify that the `create_pull_request` method is called with the correct parameters (source branch, target branch `main`, and a comprehensive AI-generated PR body), proving the successful termination of the autonomous development lifecycle.
