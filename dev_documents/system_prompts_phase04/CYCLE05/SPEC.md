# CYCLE05 Specification

## Summary
The primary objective of CYCLE05 is to implement Phase 2: Dynamic Execution within the `uat_usecase.py` service. This cycle addresses the most critical flaw in the previous architecture: the "assumed success" model, where UAT passage was simulated via a hardcoded bypass (`# Assume UAT passes for now`). This cycle replaces that placeholder with an asynchronous, fully functional, and robust integration of the Pytest and Playwright execution suite. The goal is to construct a resilient pipeline that dynamically triggers the UATs within the sandbox, rigorously captures the execution state (including discovering the multi-modal artifacts generated in CYCLE04), and formally dictates whether the system should proceed to Pull Request creation or forcibly route the workflow to the Stateless Auditor for diagnostic recovery based entirely on deterministic execution results, cementing the mechanical blockade as the ultimate arbiter of quality.

## System Architecture
The system architecture for this cycle centers entirely around the comprehensive overhaul of the `src/services/uat_usecase.py` module. The core mechanism enabling this transition is the `ProcessRunner`, which will be instantiated within the use case to asynchronously execute the `uv run pytest` command, specifically targeting the `tests/uat/` directory and explicitly enabling the Playwright plugin. The architectural boundary here is crucial: `uat_usecase.py` acts as the conductor of the "Behavioral Reality Sandbox." It must initiate the test execution asynchronously to prevent blocking the main LangGraph event loop, ensuring system responsiveness. Upon completion of the shell command, it must carefully parse the raw standard output, standard error, and the integer exit code. If the exit code is non-zero, the architecture dictates that the module must actively scan the designated artifacts directory for the multi-modal payloads (screenshots and traces) generated by the `conftest.py` hook. The output of this service is strictly defined by the newly enhanced `UatExecutionState` Pydantic model. If the tests pass, the state dictionary is updated to signal the LangGraph routing logic that the implementation is verified and a PR can be raised. If the tests fail, the `UatExecutionState` object, fully populated with error logs and validated artifact paths, is injected into the state, signaling a mandatory routing transition to the `auditor_usecase.py`. This complete replacement of the static bypass with deterministic, sandboxed execution is the functional linchpin of the automated QA pipeline, enabling true end-to-end verification.

## Design Architecture
The design architecture relies heavily on the `UatExecutionState` Pydantic model to manage the complex, multi-faceted data generated during a dynamic UI test failure. This model, which will extend the base execution state or dictionary structure required by LangGraph, will strictly type the properties governing the workflow's next steps. It will include the `exit_code` (integer), `stdout` (string), `stderr` (string), and critically, a rigorously typed list of `MultiModalArtifact` objects (defined in CYCLE04). By enforcing `ConfigDict(extra='forbid')`, the schema guarantees that `uat_usecase.py` cannot inject arbitrary, undocumented, or malformed data into the global LangGraph state, preserving the integrity of the state machine. The validation logic within this model will verify that any paths listed in the artifacts collection correspond to actual, readable files on disk before the state transition is permitted, preventing silent failures later in the pipeline. The primary consumer of this structured state is the LangGraph router (`routers.py`), which relies entirely and exclusively on the `exit_code` within this validated object to make its next routing decision. This Schema-First design ensures that the transition from chaotic dynamic shell execution back into the structured Python-based orchestration layer is seamless, strongly typed, and entirely deterministic, completely eliminating the possibility of hallucinated test results influencing the pipeline and guaranteeing robust graph transitions.

## Implementation Approach
The implementation plan for CYCLE05 starts with introducing the `UatExecutionState` schema to accurately map the data generated during execution, ensuring seamless integration with the LangGraph state dictionary. The primary execution method within `src/services/uat_usecase.py` is entirely rewritten to permanently remove the hardcoded bypass. In its place, we instantiate the `ProcessRunner` and configure it to run tests dynamically via a configurable command (e.g., `uv run pytest tests/uat/`) configured through a new `UATConfig` within `settings`, explicitly appending arguments like Playwright configuration vectors safely parsed via `shlex`.

Crucially, **strict security boundaries** have been introduced into the architectural blueprint during implementation:
- **Command Injection Prevention**: A strict whitelist algorithm explicitly validates execution binaries (`uv`, `pytest`, `python`) and blocks any arguments containing shell metacharacters before execution.
- **Path Traversal Enforcement**: The multimodal artifact scanner explicitly validates `test_id` structures using regular expressions, while directory resolution mandates strict boundary confinement, securely resolving absolute targets and explicitly handling missing zip generation cases cleanly without forcing dummy file creations.
- **Environment & State Hardening**: System configurations (like `.env` loading logic and LangGraph state fields) undergo immediate, strict boundary and whitelist checking during initialization, raising deterministic exceptions upon violation.

The asynchronous execution runs securely within these boundaries, and the resulting exit code, stdout, and tracebacks are captured subject to configurable byte-limits. A resilient, non-destructive file scanner checks the designated artifacts directory. If a non-zero exit code is detected, the method instantiates the `UatExecutionState` model containing multimodal test data (images/traces). Finally, conditional routing in `src/nodes/routers.py` uses this explicitly typed structure, branching reliably to the auditor node upon failure, solidifying the UAT pipeline as the final, immutable decision point.

## Test Strategy

### Unit Testing Approach
The unit testing strategy for this cycle focuses intensely on the isolated behavior of the rewritten `uat_usecase.py`. We will use `unittest.mock.patch` to heavily mock the `ProcessRunner` execution and the `os` file system operations, preventing the tests from actually executing full browser suites. We will simulate scenarios where `ProcessRunner` returns a successful exit code (0), verifying that the use case correctly constructs a passing `UatExecutionState` without attempting to scan for non-existent artifacts. We will then simulate a failure (non-zero exit code), mocking the file system to return a list of dummy screenshot paths. The tests must assert that the use case successfully instantiates the `UatExecutionState` containing the exact error output and the validated list of artifact paths. We will rigorously test the error handling; for instance, if the process fails but no artifacts are found on disk (e.g., a backend failure), the use case must gracefully handle this condition and still produce a valid error state without the image paths, rather than crashing. All mocked objects and return values must be explicitly type-hinted to pass the strict `mypy` requirements, ensuring the orchestrator logic is robust against shell irregularities.

### Integration Testing Approach
The integration testing strategy for CYCLE05 requires verifying the complex interaction between the `uat_usecase.py`, the `ProcessRunner`, and the LangGraph state dictionary in a controlled environment. We will construct a temporary directory containing both a perfectly passing and a deliberately failing UAT script. We will programmatically invoke the `uat_usecase.py` on this directory. For the passing script, we assert that the resulting state dictionary correctly reflects a `0` exit code, permitting the PR creation phase. For the failing script, we must verify that the actual Pytest execution occurred via the sandbox, that the `conftest.py` hook successfully generated a physical artifact, and crucially, that the `uat_usecase.py` successfully discovered that artifact on disk, populated the `UatExecutionState`, and updated the graph state dictionary with the non-zero exit code. This end-to-end integration test is paramount, as it proves that the system has successfully replaced the "assumed success" placeholder with a fully functional, multi-modal, mechanical gatekeeper capable of routing based on reality, thereby realizing the core value proposition of the new architecture.
