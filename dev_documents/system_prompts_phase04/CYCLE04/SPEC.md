# CYCLE04 Specification

## Summary
The primary objective of CYCLE04 is to implement Strategy 3 of Phase 2: The Outer Loop (Behavioral Reality Sandbox), specifically focusing on Playwright Multi-Modal Capture. This cycle represents a pivotal transition in the UAT pipeline, moving from purely structural verification (linting, backend unit tests) to dynamic, interactive behavioral validation. By tightly integrating the `pytest-playwright` plugin into the existing test suite, the system gains the crucial ability to execute end-to-end user journeys within a headless browser environment. Crucially, when these UI tests inevitably fail, the system will automatically trigger a capture sequence to record rich, multi-modal artifacts—including high-resolution full-page screenshots, comprehensive DOM traces, and browser console logs. These visual and structural snapshots serve as definitive, undeniable proof of failure, providing the Stateless Auditor agent with the exact, objective context needed to diagnose complex Human-Centered Design (HCD) defects without hallucination or guesswork, fundamentally enhancing the pipeline's diagnostic capabilities.

## System Architecture
The system architecture for this cycle focuses on extending the capabilities of the Pytest execution environment through the strategic, localized integration of the `pytest-playwright` plugin. The primary architectural modifications will occur within `tests/conftest.py`, where custom Pytest lifecycle hooks—specifically the `pytest_runtest_makereport` hook—will be implemented to dynamically intercept test failures as they occur. The architecture dictates that the Playwright browser context must be managed deterministically and securely, ensuring isolated execution for each test scenario to prevent state leakage or cross-test contamination. When a failure is detected by the custom hook, the system will immediately instruct the active Playwright `page` object to capture a high-resolution screenshot and extract the relevant DOM state. To maintain structural integrity and project organization, these generated artifacts must be saved to a designated, standardized directory (e.g., `dev_documents/artifacts/`), utilizing a rigorous, predictable naming convention based on the specific test node ID and timestamp. This centralized artifact repository serves as the critical integration boundary between the Sandbox Execution phase and the subsequent Auditor Analysis phase. The implementation must guarantee that the artifact generation process is highly robust and fails gracefully; if a screenshot cannot be captured due to a catastrophic browser crash or timeout, the pipeline must still capture the stack trace and proceed with the error state, rather than hanging indefinitely. This architectural approach guarantees that every UI failure produces a tangible, visual record, transforming abstract error strings into actionable diagnostic data.

## Design Architecture
The design architecture for CYCLE04 relies on formally defining the concept of a multi-modal test failure within the system's data structures. We will introduce a new, rigorous Pydantic schema, `MultiModalArtifact`, to structure and validate the metadata associated with a failed Playwright test. This schema will encapsulate critical information such as the absolute file paths to the locally generated screenshot (`screenshot_path`) and the DOM trace (`trace_path`), alongside the raw browser console logs and the originating Pytest failure traceback. By defining this schema with `ConfigDict(extra='forbid')`, we enforce a strict, unyielding contract: any module downstream (specifically, the Auditor agent in future cycles) that consumes these artifacts must receive perfectly structured, validated data, free from unexpected fields. The schema will include active validation logic to verify that the file paths actually exist on the local file system and are readable before the object is instantiated. This prevents "file not found" errors or broken references during the Auditor's critical analysis phase. This schema-first design ensures that the transition from the chaotic reality of a broken UI test to the structured, predictable input required by an advanced LLM is handled deterministically and safely, maintaining the integrity of the data pipeline and providing a reliable foundation for the self-healing loop.

## Implementation Approach
The implementation plan for CYCLE04 begins with updating the project dependencies in `pyproject.toml` to explicitly include `pytest-playwright` and ensuring scripts are in place to install the necessary browser binaries (e.g., chromium) within the execution environment. Following dependency installation, we will define the `MultiModalArtifact` Pydantic schema to strictly type the output of a failed UI test, accompanied by thorough unit tests validating its constraints. The core development effort then shifts exclusively to `tests/conftest.py`. We will implement the `pytest_runtest_makereport` hook to intercept the test execution lifecycle. Within this hook, we will add conditional logic to check if the current test item failed and if it utilized a Playwright `page` fixture. If both conditions are met, the hook will securely trigger the artifact capture sequence: saving a screenshot and the DOM state to the standardized `artifacts/` directory, handling potential browser timeouts gracefully. Crucially, the absolute paths to these newly created files will be injected into the test report object or a safe, custom global state mechanism, allowing the overarching `ProcessRunner` (managed by `uat_usecase.py`) to retrieve them after the Pytest suite completes its run. The final step involves writing targeted unit and integration tests to verify that the hook correctly identifies failures, reliably captures the images without throwing secondary exceptions, and successfully validates the resulting file paths against the Pydantic schema. This careful, iterative approach guarantees stability within the deeply embedded Pytest hook system.

## Test Strategy

### Unit Testing Approach
The unit testing strategy for this cycle focuses intensely on isolating the custom Pytest hook logic within `conftest.py`. We will utilize the `unittest.mock` library to simulate various Pytest execution states, specifically mocking the `pytest_runtest_makereport` lifecycle event and its associated report objects. Tests will be constructed to simulate both passing and explicitly failing Playwright tests. For simulated failures, we will inject a mocked Playwright `page` object and verify that the hook correctly calls the `page.screenshot()` and trace extraction methods with the appropriate arguments. We will assert that the hook constructs the correct file paths based on the test ID and successfully creates the necessary directories if they do not exist. Furthermore, we will rigorously test the `MultiModalArtifact` Pydantic schema in isolation, ensuring it correctly validates valid file paths and raises appropriate, expected exceptions when initialized with non-existent files or directories. All mock interactions will be strictly typed to comply with the project's `mypy` configuration, ensuring the artifact capture logic is resilient, type-safe, and predictable before broader integration, guaranteeing that the hook itself does not become a source of instability.

### Integration Testing Approach
The integration testing strategy for CYCLE04 involves executing a real, albeit simplified, Playwright test suite to verify the end-to-end artifact capture process within a live environment. We will create a temporary test file containing a deliberate, guaranteed UI failure (e.g., asserting the visibility of a non-existent DOM element or navigating to an invalid URL). We will then programmatically invoke Pytest on this specific file, ensuring the Playwright plugin is active. The primary assertion will be verifying that upon test completion, the specified `artifacts/` directory contains a valid, newly generated, non-empty `.png` screenshot and corresponding trace files. We will further assert that the Pytest execution summary correctly reflects the failure and that the custom hook did not introduce any secondary errors or unhandled exceptions that obscured the original test failure. This integration test provides absolute, empirical confirmation that the UAT pipeline can successfully transition from abstract code execution to capturing tangible, visual evidence of UI defects, forming the critical bridge to the Auditor phase.
