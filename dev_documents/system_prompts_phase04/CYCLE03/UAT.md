# CYCLE03 User Acceptance Testing

## Test Scenarios

### Scenario 1: Static Analysis Blockade (Ruff/Mypy)
The objective of this scenario is to verify that the mechanical gatekeeper successfully blocks execution and routes the error trace back to the worker when static analysis fails. The test simulates a developer or agent committing code with intentional Mypy type errors (e.g., passing a string to a function expecting an integer) and Ruff style violations (e.g., unused imports). The user initiates the cycle execution via the command-line interface. The system must invoke the `ProcessRunner` to execute `ruff check .` and `mypy .`. Because of the intentional errors, these processes will return non-zero exit codes. The expected behavior is that the pipeline halts progression immediately, successfully parses the stderr outputs into the `StructuralGateReport`, and deterministically routes the session back to the coding agent, providing it with the exact linting and typing error logs. This interactive Marimo test ensures the system refuses to advance broken code based on rigorous static analysis, visually proving the effectiveness of the mechanical blockade.

### Scenario 2: Dynamic Unit Test Blockade (Pytest)
This scenario validates the system's ability to mechanically block execution when dynamic Pytest unit tests fail, even if static analysis passes. The test simulates a codebase where all Ruff and Mypy checks pass perfectly, but a structural unit test explicitly fails due to incorrect business logic or a failed assertion. The user executes the implementation cycle. The system must execute `uv run pytest` via the `ProcessRunner`. Upon receiving a non-zero exit code from Pytest, the orchestration layer must package the standard error traceback and route it back to the active session of the Jules agent for a rapid fix. The pipeline must unequivocally block any attempt to finalize the cycle, generate documentation, or create a Pull Request. This Marimo scenario visually confirms that deterministic, dynamic test execution serves as the ultimate gatekeeper, entirely replacing the subjective and flawed LLM-based code review process and enforcing strict TDD principles.

## Behavior Definitions
GIVEN the Coder agent generates code containing static type errors or syntax violations, WHEN the execution pipeline reaches the evaluation phase, THEN the mechanical gatekeeper must execute `ruff` and `mypy`, capture the resulting non-zero exit codes, AND immediately route the execution trace back to the active worker session, absolutely preventing the creation of a Pull Request.

GIVEN the generated code passes all static checks but fails a structural unit test, WHEN the `ProcessRunner` executes the Pytest suite, THEN the system must capture the test failure traceback, structure it within the defined Pydantic schema, AND redirect the workflow back to the Coder agent for remediation, ensuring that only code achieving a 100% pass rate across all checks is permitted to advance through the automated UAT pipeline. This behavior definition forms the core logic of the Phase 1 backend verification gate, ensuring that no compromised structural code ever reaches the multi-modal outer loop.
