# CYCLE06 Specification

## Summary
The primary objective of CYCLE06 is to implement Phase 3: The Evaluation, Recovery, and Tracing Loop, focusing entirely on the Stateless Auditor and its deep integration with OpenRouter and LangSmith. This cycle represents the culmination and crowning achievement of the UAT pipeline, establishing the critical "Outer Loop" diagnostician. When dynamic UAT execution inevitably fails within the sandbox, this phase is responsible for securely piping the captured multi-modal artifacts (screenshots, error logs) to an advanced Vision LLM. The core requirement is to enforce an adversarial double-check sequence via specialized prompting to prevent false positives and, most importantly, to extract a surgical, schema-compliant JSON `FixPlanSchema` from the Auditor. This highly structured plan is then deterministically routed back to the Worker's active session, completing the autonomous self-healing loop while ensuring total, unquestionable observability of the prompt, response, and graph transitions, fundamentally solving the "Black Box" issue.

## System Architecture
The system architecture for this final cycle focuses on the `src/services/auditor_usecase.py` (or a significant enhancement of the existing `qa_usecase.py`) and its interactions with the external OpenRouter API. The architecture strictly mandates stateless execution for the Auditor component to prevent context dilution. It receives the `UatExecutionState` containing the error context and physical artifact paths. The module will securely encode the screenshots (e.g., via base64 encoding) and construct a highly specific, multi-modal prompt defined in a dedicated system prompt template. This prompt will enforce the "Devil's Advocate" adversarial reasoning to ensure the LLM rigorously evaluates whether the failure is a genuine code defect or merely a flaky test artifact. Crucially, the external API call must be meticulously traced using LangSmith decorators or context managers, ensuring that the raw image payload, the prompt text, the exact JSON response, and the latency are fully and permanently logged. Once the response is received, it must be strictly parsed and validated against the `FixPlanSchema`. Only a fully validated JSON plan is injected back into the LangGraph state dictionary. The graph router then directs the workflow back to the Stateful Worker (`coder_usecase.py`), which simply applies the structured diff or instructions provided in the `FixPlanSchema`, effectively resolving the bug without needing the entire project context, creating an incredibly resilient, closed-loop development environment.

## Design Architecture
The design architecture for CYCLE06 relies heavily on the finalization and strict enforcement of the `FixPlanSchema` Pydantic model. This schema represents the authoritative, unyielding contract between the Stateless Auditor and the Stateful Worker. It must define strictly typed fields for `target_file` (the file to edit), `defect_description` (the reasoning), and `git_diff_patch` (or an equivalent precise structural modification instruction block). The `ConfigDict(extra='forbid')` configuration is essential to prevent the Auditor from hallucinating extra instructions, commentary, or malformed data that could crash the Worker or corrupt the repository. The integration point is critical and carefully managed: `auditor_usecase.py` consumes the `UatExecutionState`, interacts with the external, unpredictable LLM, and forces the output into the predictable `FixPlanSchema`. The LangGraph state dictionary must be updated to cleanly hold this `FixPlanSchema` so that `routers.py` can successfully transition the graph back to the Worker. This schema-first approach completely isolates the unpredictability of the Vision LLM, ensuring that only verified, actionable, and syntactically valid recovery plans are permitted to influence the source code. The tracing requirements (LangSmith) do not strictly alter the domain models but dictate how the services interact with the execution context, demanding explicit configuration passing during the API invocation to ensure observability, proving that the fix originated from a deterministic, traceable process.

## Implementation Approach
The implementation plan begins with finalizing the `FixPlanSchema` within the `domain_models` directory, ensuring rigorous unit tests validate its constraints and parsing capabilities. Next, a new, detailed system prompt template (`UAT_AUDITOR_INSTRUCTION.md`) will be created in `src/templates/`. This template explicitly defines the adversarial double-check methodology and mandates the JSON output format matching the schema. Development then focuses on `src/services/auditor_usecase.py`. This service will read the `UatExecutionState`, encode the referenced multi-modal artifacts into base64, and invoke the OpenRouter API. The raw string response will be parsed and validated against `FixPlanSchema`. A crucial step is integrating LangSmith: ensuring that the `LANGCHAIN_TRACING_V2` environment variable is respected and that the specific API call within the Auditor use case is properly decorated or context-managed to log the massive full payload. Finally, the LangGraph `routers.py` and the `coder_usecase.py` (Worker) will be updated to handle the incoming `FixPlanSchema`, applying the surgical fix and restarting the execution cycle loop. Extensive testing will confirm that the graph correctly visualizes the transition from Auditor back to Coder, ensuring the self-healing mechanism operates flawlessly under live conditions.

## Test Strategy

### Unit Testing Approach
The unit testing strategy for this cycle is intensely focused on the parsing, validation, and error handling logic within the `auditor_usecase.py`. We will utilize `unittest.mock.patch` to completely mock the external OpenRouter API response, simulating both perfectly formed, schema-compliant JSON strings and malformed or wildly hallucinated responses from the LLM. The tests must assert that `auditor_usecase.py` successfully instantiates a `FixPlanSchema` from valid JSON and handles validation errors gracefully (e.g., retrying the prompt or failing securely) when presented with invalid data, preventing system crashes. We will also test the base64 encoding logic for the multi-modal artifacts, ensuring that valid file paths result in correctly encoded strings suitable for the Vision LLM payload. All mocked parameters, especially the mock API responses, will be explicitly typed to satisfy `mypy` requirements. These isolated unit tests guarantee that the system's internal boundaries are secure against unpredictable LLM outputs, ensuring the outer loop diagnostician is robust.

### Integration Testing Approach
The integration testing strategy for CYCLE06 involves simulating a complete, end-to-end self-healing loop within the LangGraph orchestration environment. We will construct a temporary environment with a known, fixable UAT failure. We will mock the external OpenRouter API to return a predefined, correct `FixPlanSchema` corresponding to the known failure. We will then programmatically execute the graph. The critical assertions are: first, that the graph correctly routes from the execution node to the Auditor node; second, that the Auditor node correctly parses the mock and updates the graph state with the `FixPlanSchema`; third, that the router successfully directs the state back to the Coder node; and finally, that the Coder successfully applies the fix to the file. We will also utilize LangSmith's testing utilities or manual verification during development to ensure that the trace of this complete loop, including the state dictionary mutations and the mocked API call, is successfully recorded and visible. This end-to-end integration test is the definitive proof that the automated UAT pipeline can deterministically diagnose and recover from dynamic execution failures autonomously.
