# CYCLE06 User Acceptance Testing

## Test Scenarios

### Scenario 1: Structured Fix Plan Generation
The objective of this scenario is to verify the system's ability to extract a highly structured, surgical `FixPlanSchema` from a failed UI test and route it correctly. The test simulates a known UI failure that generates an error trace and a screenshot. The execution pipeline is triggered, and the system intercepts the failure, invoking the `auditor_usecase.py`. The OpenRouter API is (either mocked or utilized with a deterministic prompt) to return a valid JSON fix plan. The expected behavior is that the system strictly validates this JSON against the Pydantic schema, successfully injecting the `FixPlanSchema` into the global state dictionary, and visually demonstrating (via console logs or the LangGraph visualizer) the routing transition back to the Worker session for remediation. This interactive Marimo scenario confirms the core diagnostic capability, proving the Auditor can translate visual chaos into structural order.

### Scenario 2: LangSmith Observability Verification
This scenario validates the "Panopticon" Observability Layer. The test involves executing a complete, failing UAT loop (Execution -> Auditor -> Coder). The user initiates the cycle with LangSmith variables active. The expected outcome is not just the successful routing of the fix plan, but the definitive generation of a complete trace within the LangSmith UI. The developer must be able to visually verify the graph node transitions (proving no infinite loops occurred silently), inspect the exact mutation of the state dictionary (e.g., verifying `current_fix_plan` was populated), and crucially, view the raw multi-modal prompt (including the base64 screenshot) and response latency of the Auditor LLM call. This Marimo scenario guarantees that the system is fully observable and that the "Black Box" has been permanently eliminated.

## Behavior Definitions
GIVEN the dynamic UAT execution fails AND the Auditor is invoked with multi-modal artifacts, WHEN the external Vision LLM returns a structured diagnostic response, THEN the system must strictly validate the response against the `FixPlanSchema`, inject the validated plan into the state dictionary, AND definitively route the execution trace back to the Coder agent for immediate, context-retained remediation.

GIVEN the complete diagnostic and recovery loop executes, WHEN the developer inspects the LangSmith tracing dashboard, THEN they must be able to visualize the explicit node routing graph, examine the precise state mutations between the UAT, Auditor, and Coder nodes, AND verify the raw multi-modal prompts and responses sent to the external models, ensuring absolute transparency and accountability of the automated QA pipeline. This ensures that every self-healing action is fully documented and auditable by human developers.
