# CYCLE06 SPECIFICATION

## Summary
This cycle implements Phase 2: The Outer Loop (Behavioral Reality Sandbox) focusing entirely on Dynamic Execution within `src/services/uat_usecase.py`. The fundamental objective is to permanently replace the legacy '# Assume UAT passes for now' placeholder with a dynamic, reality-based execution of the complete Pytest/Playwright suite. This enforces the mechanical gatekeeper, ensuring that the Stateful Worker's generated code actually performs the requested behaviors in a simulated browser environment before advancing to pull request creation. This phase transitions the NITPICKERS framework from theoretical code generation to verifiable software engineering. By delegating execution to the `ProcessRunner` established in earlier cycles, the orchestration layer guarantees that success is proven, not assumed. Furthermore, the cycle integrates the multi-modal artifacts captured during CYCLE05 into the Pydantic domain models defined in CYCLE01, creating a structured, serialized payload that perfectly describes the failure context for downstream consumption by the Stateless Auditor.

## System Architecture
The system architecture for this cycle introduces the dynamic outer loop execution model, completely overhauling the `UatUseCase` service. Rather than relying on static assertions or basic unit tests, `src/services/uat_usecase.py` delegates execution to the asynchronous `ProcessRunner`. It orchestrates the asynchronous execution of `uv run pytest tests/` with the playwright capture hooks active.

The core architectural flow dictates that the LangGraph workflow routes the application state to the `UatUseCase`. The service initializes the `ProcessRunner`, awaits the command execution, and captures the combined standard output and error streams. The result of this execution maps directly to the `uat_models.py` schema (e.g., `UATResult`).

If the `ProcessRunner` returns an exit code of `0`, the pipeline concludes that all behavioral tests have passed. The state dictionary is updated to reflect success, and the LangGraph orchestrator advances linearly to the PR creation node.

However, if the `ProcessRunner` returns a non-zero exit code (e.g., `1`), the architecture dictates a complex state mutation. The `UatUseCase` must aggressively parse the `dev_documents/test_artifacts/` directory to locate the physical `.png` screenshots and `.txt` DOM traces generated by the Playwright hooks (CYCLE05). These file paths, along with the captured error trace, are bundled into the `UATResult` Pydantic model. This structured state vector is then explicitly routed back to the Stateless Auditor node, providing irrefutable visual and structural evidence of the failure. This strict boundary management ensures that the Stateful Worker is never blindly trusted.

## Design Architecture
The design architecture relies heavily on robust execution logic and file system traversal within `uat_usecase.py`. The `UatUseCase` class must encapsulate the orchestration logic, translating raw command-line execution results into structured Pydantic state dictionaries required by the LangGraph state machine.

Key design invariants include:
1.  **Asynchronous Execution**: The `UatUseCase` must leverage the asynchronous nature of the `ProcessRunner` to avoid blocking the main event loop while the potentially long-running end-to-end UI tests execute.
2.  **Robust Parsing**: The service must parse the standard output and standard error deterministically. It must extract relevant traceback information to populate the `stderr` field of the `UATResult` model.
3.  **Deterministic File Resolution**: Upon failure, the service must reliably locate the generated screenshot paths from the designated output directory (`dev_documents/test_artifacts/`). This requires globbing or specifically targeting files prefixed with the failing test names parsed from the Pytest output.
4.  **State Mutation**: The service is responsible for updating the global LangGraph state dictionary with the newly instantiated `UATResult` object. The `uat_exit_code` must be explicitly set so that the downstream router nodes (`src/nodes/routers.py`) can effectively evaluate the conditional edges.

## Implementation Approach
The implementation approach involves comprehensively rewriting `src/services/uat_usecase.py`.

**Step 1:** Remove the legacy hardcoded pass logic (`# Assume UAT passes for now`).

**Step 2:** Inject the `ProcessRunner` dependency into the `UatUseCase`.

**Step 3:** Implement the `execute_uat` asynchronous method. Construct the shell command (e.g., `uv run pytest tests/`) and `await` the `ProcessRunner.run()` method. Capture the `exit_code`, `stdout`, and `stderr`.

**Step 4:** If the `exit_code` is `0`, instantiate a successful `UATResult` and update the state.

**Step 5:** If the `exit_code` is non-zero, implement a traversal routine over `dev_documents/test_artifacts/`. Identify the most recently generated `.png` and `.txt` files corresponding to the failed test. Instantiate the `UATResult` Pydantic model containing the exit code, logs, and absolute paths to these artifacts.

**Step 6:** Return the mutated state dictionary, ensuring it perfectly aligns with the expected LangGraph state schema, enabling seamless routing to the Stateless Auditor.

## Test Strategy

### Unit Testing Approach
The unit tests will extensively use `unittest.mock.patch` to isolate the `UatUseCase` from actual shell execution and file system I/O. We will mock the `ProcessRunner.run()` method. We will simulate a successful run returning exit code `0` and a failed run returning exit code `1`. For the failure scenario, we will mock `os.listdir` or `pathlib.Path.glob` to return mock artifact paths. We will verify that the `UatUseCase` correctly evaluates the exit code, locates the mock artifacts, and accurately instantiates the `UATResult` Pydantic model with the correct data types and strings. We will assert that exceptions during file traversal (e.g., missing artifacts despite a non-zero exit code) are handled gracefully, perhaps by returning a `UATResult` with empty path strings but a preserved error trace.

### Integration Testing Approach
The integration testing approach will invoke the `UatUseCase` in a heavily mocked application context containing a simple, pre-configured Pytest script. We will configure one script that passes perfectly and another script that fails intentionally (e.g., `assert False`). We will execute the use case and assert that it accurately runs the script via subprocess, captures the non-zero exit code, automatically traverses the test directory to locate the generated screenshot (simulated by a touch command), and correctly populates the state dictionary with the absolute artifact paths. This proves the dynamic execution and artifact bundling logic functions perfectly as the central UAT gatekeeper.
