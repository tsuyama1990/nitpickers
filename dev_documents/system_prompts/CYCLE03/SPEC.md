# CYCLE03 Specification

## Summary
The primary objective of CYCLE03 is to implement Strategy 2 of Phase 1: Backend Verification and Gatekeeping. This cycle is critical for establishing the unyielding "Mechanical Blockade" that guarantees structural integrity before any Pull Request creation or outer-loop diagnostic processing occurs. It leverages the existing, sandboxed `ProcessRunner` to execute unit tests and static analysis tools locally, dynamically parsing their exit codes to explicitly and deterministically prevent broken code from advancing through the pipeline. By mechanically enforcing strict Type Checking (via `mypy`) and Linting (via `ruff`), and directly coupling any test failures to the Worker's session recovery mechanism, this cycle completely eliminates the reliance on subjective LLM code reviews. It replaces the flawed "assumed success" model with a deterministic, unyielding validation gate, ensuring only code meeting the highest professional standards progresses to the behavioral testing phase, fundamentally shifting the paradigm from probabilistic LLM assessment to deterministic execution verification.

## System Architecture
The system architecture for this cycle revolves around significantly enhancing the orchestration nodes that manage the critical transition between code generation and evaluation within the LangGraph workflow. The primary integration points are located within `src/nodes/sandbox_evaluator.py` (or the equivalent evaluation routing logic) and `src/services/uat_usecase.py`. The implementation will utilize the `ProcessRunner` to asynchronously execute three distinct, mandatory verification commands within the secure sandbox: `uv run ruff check .`, `uv run mypy .`, and `uv run pytest`. The architectural boundary established here is strict and uncompromising: the output of these commands must be captured and parsed, and any non-zero exit code must trigger a deterministic routing decision. Instead of proceeding to the "Outer Loop" Auditor, marking the task complete, or attempting to generate a Pull Request, the architecture dictates that the exact standard error (stderr) trace be packaged into a highly structured error report. This report is then routed directly back to the active session of the Stateful Worker (the Coder agent). This precise feedback loop ensures that Jules receives immediate, objective, and actionable feedback without the cognitive overhead or context dilution associated with full project re-evaluation. The orchestration logic will be carefully modified to enforce this blockade, effectively making the evaluation phase an impenetrable gatekeeper that guarantees 100% structural pass rates before advancing. This architecture is designed to handle failure states gracefully, ensuring that syntax errors in the generated code do not crash the agentic framework itself, but rather trigger a controlled, self-healing loop.

## Design Architecture
The design architecture for CYCLE03 relies on a Schema-First definition to formalize the verification feedback loop. We will introduce the `VerificationResult` Pydantic model to rigorously structure and type the output of the `ProcessRunner` executions. This schema will encapsulate vital data points: the specific command executed, the integer exit code returned by the shell, the full standard output (`stdout`), and the critical standard error traceback (`stderr`). The `VerificationResult` model will enforce invariants, such as ensuring the exit code is strictly an integer and that the stderr string is safely sanitized for downstream consumption. Building upon this, a higher-level model, `StructuralGateReport`, will aggregate multiple `VerificationResult` objects—specifically one for linting, one for type checking, and one for testing—providing a comprehensive, single-object snapshot of the system's structural integrity. This aggregated report will be consumed directly by the LangGraph routing logic. By utilizing `ConfigDict(extra='forbid')`, we ensure that the routing mechanism receives only perfectly structured data, preventing unpredictable state transitions caused by malformed execution logs. The design strictly isolates the dynamic execution of external commands within the secure, local sandbox environment, ensuring that the main agent orchestration loop remains stable, secure, and entirely unaffected by syntax errors or infinite loops within the generated code itself. This robust, typed design is crucial for replacing subjective evaluation with definitive, mechanical truth, providing an unshakable foundation for the UAT pipeline.

## Implementation Approach
The implementation plan for CYCLE03 commences with defining the `VerificationResult` and `StructuralGateReport` Pydantic schemas within the `domain_models` directory. Once these schemas are thoroughly validated via unit tests to ensure strict invariant enforcement, development will focus on the service layer, specifically enhancing the `ProcessRunner` interactions within the evaluation use cases. We will implement dedicated asynchronous methods to sequentially and securely execute the Ruff linting, Mypy type checking, and Pytest suites. The results of these executions will be captured and strictly parsed into the newly defined Pydantic models. Crucially, we will update the LangGraph node logic, specifically within `sandbox_evaluator.py` or `routers.py`, to consume the `StructuralGateReport`. The routing logic will be updated: if any underlying `VerificationResult` indicates a non-zero exit code, the workflow must be mechanically blocked. The node logic will package the failing stderr trace and explicitly route the graph execution back to the `Coder` node, supplying the error report directly into Jules's active session for immediate, context-aware remediation. The final phase involves thorough testing of this new routing logic, ensuring that valid code passes the gate smoothly while invalid code deterministically triggers the recovery loop, solidifying the mechanical blockade. This iterative, test-driven implementation guarantees that the gatekeeper functions exactly as designed before being integrated into the main application flow.

## Test Strategy

### Unit Testing Approach
The unit testing strategy for this cycle is designed to meticulously validate the precision of the new routing logic and the absolute integrity of the Pydantic schemas in isolation. Comprehensive unit tests will be written for the `VerificationResult` and `StructuralGateReport` models to ensure they correctly parse process outputs and rigidly enforce all defined invariants, rejecting malformed data. The evaluation node logic will be rigorously tested by injecting mock `ProcessRunner` objects that simulate various complex combinations of successful (exit code 0) and failing (non-zero exit code) linting, type checking, and Pytest executions. These tests will forcefully assert that the routing logic makes the correct deterministic decisions—specifically, that it correctly formats the error report and triggers a state transition back to the `Coder` node when failures occur, and proceeds only when all checks pass. Furthermore, tests will verify that the system correctly parses complex traceback strings without crashing or misinterpreting the error source. By extensively utilizing dependency injection and meticulously type-hinted mocks, these unit tests will guarantee that the mechanical blockade functions flawlessly under all simulated failure conditions before being deployed in the live orchestration environment. This approach is fundamental to ensuring that the gatekeeper remains reliable, even when processing highly erratic output from generated code.

### Integration Testing Approach
The integration testing strategy for CYCLE03 focuses on confirming the end-to-end functionality of the mechanical blockade within a live, albeit controlled, LangGraph execution flow. We will set up a temporary test environment containing deliberate, specific structural flaws—such as severe Python syntax errors, explicit Mypy type violations, and failing unit tests. We will programmatically initiate the LangGraph workflow and assert that the system successfully executes the respective verification commands via the real `ProcessRunner` in the sandbox. Crucially, the tests will verify that the workflow execution is deterministically halted at the evaluation node, that the resulting error traces are accurately and completely captured within the `StructuralGateReport`, and that the pipeline explicitly refuses to proceed to the Pull Request generation phase. Conversely, we will run integration tests on a perfectly clean, valid codebase to ensure the blockade does not yield false positives and successfully permits the pipeline to advance. These end-to-end tests provide the final, undeniable validation that the "assumed success" paradigm has been completely eradicated, replaced by a robust, mechanical verification process that strictly adheres to the project's quality standards.
