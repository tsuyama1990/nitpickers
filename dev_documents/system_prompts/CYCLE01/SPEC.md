# CYCLE01 SPECIFICATION

## Summary
This implementation cycle focuses entirely on establishing the foundational data structures and domain models required for the automated User Acceptance Testing (UAT) pipeline. In modern agentic frameworks, the robustness of the system is directly correlated with the strictness of its state management and data serialization boundaries. The primary goal of CYCLE01 is to define Pydantic-based schemas that will act as the authoritative contract across the entire NITPICKERS workflow. This includes the Worker, the Auditor, and the Observability Layer.

Specifically, we need to design models that can reliably encapsulate multi-modal artifacts (such as paths to captured DOM states and full-page screenshots), capture strict exit codes from structural testing tools (like `ruff` and `mypy`), and formalize the surgical "Fix Plan" generated by the OpenRouter Vision LLM. These models must seamlessly integrate into the pre-existing project state dictionary without causing breaking changes or regressions in the current CLI execution loop. By enforcing strict typing at the very beginning of the project, we ensure that subsequent cycles—which will implement the complex logic for parsing Pytest hooks, executing sandboxed Playwright scripts, and routing state between LangGraph nodes—can rely on a deterministic, fully validated data layer. This schema-first approach is the cornerstone of the zero-trust validation model, guaranteeing that malformed data or LLM hallucinations are caught mechanically before they can pollute the execution context.

## System Architecture
The system architecture for this cycle revolves around the robust definition of domain objects using Pydantic. These schemas enforce strict type safety and guarantee that all internal states—particularly those relating to the newly introduced UAT phase—are meticulously validated before being processed by any service or workflow node.

The architecture dictates that the core project state (`src/domain_models/state.py`) must be extended. It must hold new references to dynamically generated test results, specifically tracking the `uat_exit_code` and any associated multi-modal debugging payloads. The integration of these models serves as the critical interface between the "Inner Loop" (the Stateful Worker's structural verification) and the "Outer Loop" (the dynamic Playwright execution and the Stateless Auditor's diagnosis).

The `uat_models.py` file will define the structures for capturing visual testing artifacts. When a UI test fails, the Playwright hooks (developed in later cycles) will populate this model with paths to the `.png` screenshot and the `.txt` DOM trace. This explicit structuring ensures the Auditor node always receives the correct context.

Furthermore, the `fix_plan.py` file introduces a highly constrained schema designed specifically for the Vision LLM's output. To prevent the "Black Box" issue where an agent rewrites large swathes of code unpredictably, the Fix Plan model enforces surgical, precise instructions (e.g., specific file paths, search strings, and replacement blocks). The LangGraph state machine will rely entirely on the successful validation of this JSON object before allowing the workflow to route back to the Worker node for implementation.

## Design Architecture
The design architecture is strictly schema-first, leveraging the full capabilities of Pydantic. We are expanding `src/domain_models/state.py` and creating entirely new, isolated modules (`uat_models.py` and `fix_plan.py`) to maintain a clean separation of concerns.

1.  **UAT Artifact Models (`uat_models.py`)**: This model represents the physical outputs of a failed dynamic test. It must define optional string fields for `screenshot_path`, `dom_trace_path`, and `console_logs`. A key invariant is that if the `uat_exit_code` (which will be tracked in the main state) indicates a failure, these paths must resolve to valid files in the local filesystem before the payload is sent to the Auditor.
2.  **Surgical Fix Plan (`fix_plan.py`)**: This model defines the structure of the LLM's response. It must contain a list of modification objects, each requiring a `filepath`, an `explanation` for the change, and the specific `code_snippet` or `search_replace_block` to be applied. The invariant here is strict validation: the LLM must return JSON that perfectly matches this schema, preventing arbitrary conversational text from breaking the parsing logic.
3.  **Extended State (`state.py`)**: The existing global state dictionary utilized by LangGraph must be updated. We will add `uat_exit_code` (an integer, defaulting to 0), `uat_artifacts` (an optional instance of the UAT Artifact model), and `current_fix_plan` (an optional instance of the Fix Plan model). These additions must be designed for backward compatibility, ensuring existing agents that do not utilize the UAT pipeline can still function normally.

## Implementation Approach
The implementation requires an additive mindset, focusing on extending the domain models without introducing breaking changes to legacy components.

**Step 1: Create `fix_plan.py`**
Navigate to `src/domain_models/` and create `fix_plan.py`. Define a `FileModification` Pydantic model with fields: `filepath` (str), `explanation` (str), and `diff_block` (str). Then, define the parent `FixPlan` model containing a list of `FileModification` objects. Ensure strict config is enabled (`model_config = ConfigDict(strict=True)`).

**Step 2: Create `uat_models.py`**
In the same directory, create `uat_models.py`. Define a `UATResult` model encompassing `exit_code` (int), `stderr` (str), `screenshot_path` (Optional[str]), and `dom_trace_path` (Optional[str]). Use Pydantic field validators to ensure that if a path is provided, it is a non-empty string.

**Step 3: Update `state.py`**
Carefully modify the existing `src/domain_models/state.py`. Import the newly created `UATResult` and `FixPlan` models. Add them as optional fields to the primary LangGraph State `TypedDict` or Pydantic `BaseModel` used by the project. For example: `uat_result: UATResult | None = None` and `current_fix_plan: FixPlan | None = None`. This ensures the state machine can track the lifecycle of a UAT failure and its subsequent surgical repair.

## Test Strategy

### Unit Testing Approach
The unit testing approach will rigorously validate the Pydantic schemas in isolation. Using `pytest`, we will create instances of `UATResult` and `FixPlan` with valid dictionaries to verify successful parsing. More importantly, we will write negative tests that intentionally inject malformed data (e.g., missing required fields, incorrect data types, or invalid JSON structures). We will use `pytest.raises(ValidationError)` to guarantee that the schemas act as true mechanical gatekeepers. Special attention will be given to the `FixPlan` model to ensure that nested lists of file modifications are serialized and deserialized deterministically, matching the exact format expected from the OpenRouter API.

### Integration Testing Approach
The integration testing approach will verify that the updated state dictionaries correctly interact within a mock execution context. We will simulate a state transition within a dummy LangGraph workflow. A mock node will inject a simulated failed Playwright run—populating the `uat_result` field with an exit code of 1 and dummy artifact paths. A subsequent mock node will verify that this state is retained and can be accessed without data loss. We will also test the serialization of the entire state object using `model_dump_json()` to ensure that the added UAT models do not break the logging or tracing mechanisms that rely on JSON serialization.
