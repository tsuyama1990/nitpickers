# CYCLE01 Specification

## Summary
The primary objective of Cycle 01 is to refactor the foundational state management models, particularly the `CycleState` within LangGraph, and to implement the fundamental routing logic for Phase 2. This cycle establishes the architectural bedrock for the entire 5-Phase pipeline by introducing crucial control variables that govern the flow of execution between the parallel Coder agents, the local sandbox evaluations, and the sequential Auditor nodes. The `is_refactoring`, `current_auditor_index`, and `audit_attempt_count` fields will be formally defined as strictly typed Pydantic properties. The successful implementation of this cycle ensures that the core workflow engine can deterministically navigate complex, conditional logic paths based on immutable state representations, paving the way for the robust, zero-trust evaluation mechanisms defined in subsequent cycles. The ultimate goal is to achieve seamless, predictable graph transitions without relying on fragile global variables or implicit assumptions about the codebase's current state.

## Infrastructure & Dependencies
- **A. Project Secrets (`.env.example`):** No external services or API keys are required for the implementation of the state models and routing logic in this cycle. However, the Coder must ensure that any existing `# Target Project Secrets` comments remain intact within `.env.example` to support future external integrations.
- **B. System Configurations (`docker-compose.yml`):** The implementation of pure Python state models and routing functions does not necessitate any changes to internal ports or environment setups. The Coder must explicitly preserve the existing, valid YAML formatting and idempotency of the `docker-compose.yml` file.
- **C. Sandbox Resilience (CRITICAL TEST STRATEGY):** Although this cycle primarily involves local logic, the Coder MUST explicitly ensure that *all external API calls relying on the newly defined secrets in `.env.example` MUST be mocked in unit and integration tests (using `unittest.mock` or `pytest-mock`)*. This is a non-negotiable requirement to guarantee that the sandbox evaluation can execute rapidly and reliably without attempting real network calls, thereby preventing infinite retry loops or unexpected billing surges if the generated logic attempts unauthorized access during testing.

## Design Architecture
The design of the NITPICKERS system is fundamentally anchored in Pydantic-based schemas, enforcing strict validation and predictability. The `CycleState` model represents the core domain concept of a development iteration. In this cycle, the model must be extended with key invariants: `is_refactoring` must strictly be a boolean, defaulting to `False`, to cleanly delineate the initial implementation phase from the final polishing phase. `current_auditor_index` and `audit_attempt_count` must be integers, utilizing Pydantic's `Field` constraints to enforce logical boundaries (e.g., `ge=0`). These constraints guarantee that the routing functions will always receive well-formed data. The primary consumers of this data are internal modules—specifically the LangGraph routing edges—while the producers are the various node execution handlers that mutate the state upon completion of their specific tasks. This design guarantees backward compatibility; existing functions that do not utilize the new fields will simply ignore them, while new functions can rely on their guaranteed presence and validity.

## Implementation Approach
1.  **State Model Enhancement:** Begin by thoroughly analyzing the existing `src/state.py`. Carefully inject the `is_refactoring`, `current_auditor_index`, and `audit_attempt_count` fields into the `CycleState` Pydantic model. Ensure that the `ConfigDict(extra='forbid', strict=True)` settings are strictly adhered to.
2.  **Routing Logic Implementation:** Construct the conditional routing functions within `src/nodes/routers.py`. The `route_sandbox_evaluate` function must evaluate both the `sandbox_status` and the `is_refactoring` flag to return "failed", "final_critic", or "auditor". The `route_auditor` function must handle complex fallback logic based on the `audit_attempt_count` and increment the `current_auditor_index` upon approval, returning "pass_all" when the threshold is met. The `route_final_critic` is a straightforward binary check.
3.  **Strict Typing and Linting Compliance:** Continuously execute `uv run mypy .` and `uv run ruff check .` during the implementation phase. Resolve any cyclomatic complexity warnings by extracting complex conditional logic into distinct helper functions, maintaining a maximum complexity score of 10.
4.  **Mock Integration Preparation:** Even though this cycle primarily deals with internal state, establish the testing framework using Pytest, ensuring that any potential future external dependencies are structurally mocked using `pytest-mock` to guarantee sandbox resilience.

## Test Strategy

### Unit Testing Approach (Min 300 words)
The unit testing strategy for the state models and routing logic demands absolute rigor. The Coder agent must generate comprehensive test cases utilizing Pytest to validate every conceivable combination of state variables. For the `CycleState` enhancements, the tests must assert that the Pydantic model correctly enforces the defined constraints, raising appropriate validation errors if incorrect data types are supplied. Furthermore, the unit tests for `src/nodes/routers.py` must exhaustively evaluate the conditional pathways. For example, a test must explicitly instantiate a `CycleState` object with `sandbox_status="failed"` and verify that `route_sandbox_evaluate` strictly returns "failed", regardless of the `is_refactoring` flag. Similarly, complex boundary conditions within `route_auditor`—such as exceeding the maximum `audit_attempt_count` or reaching the final auditor in the sequence—must be rigorously tested. Crucially, these tests must execute entirely in isolation, utilizing `pytest-mock` to simulate any dependencies outside of the immediate function under test. This adherence to sandbox resilience guarantees that the unit test suite remains incredibly fast and immune to environmental fluctuations, satisfying the system design objectives of zero-trust verification without incurring unnecessary API overhead.

### Integration Testing Approach (Min 300 words)
While the primary focus is on unit-level validation of the state and routers, integration testing is critical to verify that these components function harmoniously within the broader context of a LangGraph workflow. The integration tests must construct a simplified, mock version of the Coder Graph, encompassing dummy nodes for the Coder, Sandbox, and Auditor components. The primary objective is to inject a synthesized `CycleState` into this mock graph and trace its complete traversal based purely on the newly implemented routing logic. For instance, a test must assert that a state with `is_refactoring=True` and a successful sandbox evaluation correctly bypasses the standard Auditor sequence and routes directly to the Final Critic node. Furthermore, the integration tests must incorporate the mandatory DB Rollback Rule; if any state persistence mechanisms are introduced (even mocked ones), the Pytest fixtures must explicitly start a transaction before the graph execution and roll it back immediately afterward. This ensures that the simulated workflow runs do not pollute the testing environment or cause cascading failures in subsequent integration tests. This comprehensive approach guarantees that the foundational state and routing enhancements are fully integrated and ready to support the complex autonomous operations of the subsequent 5-Phase architecture cycles.
