# Cycle 01: Planning & Self-Critic Setup

## 1. Summary
Cycle 01 implements the critical "Self-Critic Node" within the existing Architect (gen-cycles) phase. The primary objective is to mathematically verify and validate the initial system architecture generated by the AI before any actual implementation cycles begin. This effectively establishes a "Red Team" dynamic within the planning phase itself. Instead of relying on a potentially flawed blueprint that will inevitably cause cascading failures during parallel development, the `gen-cycles` pipeline is extended. The Architect node now feeds its proposed `SYSTEM_ARCHITECTURE.md` and `SPEC_cycleXX.md` files into a dedicated Self-Critic node running within the identical Jules session. This node is strictly governed by a hardcoded, static checklist of anti-patterns (e.g., N+1 problems, race conditions, lack of interface contracts). If the design fails this rigorous inspection, the Architect is forced to revise and regenerate the blueprint. The cycle concludes only when the design is cryptographically locked, establishing an immutable contract that all subsequent concurrent cycles must adhere to, guaranteeing a robust foundation for massive throughput.

## 2. System Architecture
The architecture for this cycle modifies the LangGraph workflow defined in `ac_cdd_core/graph.py` and `ac_cdd_core/graph_nodes.py`. We will inject a new node, `architect_critic_node`, directly following the initial `architect_session_node`. Crucially, this node does not instantiate a new LLM session; it leverages the existing `jules_session_state.py` context to maintain continuity. The core logic involves a Python script dynamically assembling a comprehensive verification prompt based on `ARCHITECT_CRITIC_INSTRUCTION.md`, pushing this prompt into the Jules session, and parsing the response. If the response indicates vulnerabilities, the state machine routes back to `architect_session_node`.

## 3. Design Architecture
The implementation relies heavily on Pydantic to ensure the integrity of the state machine. We will extend the existing `CycleState` and internal status enums to represent the new transition states.

1.  **State Extension**: `CycleState` in `src/state.py` will be subtly extended to include a `critic_feedback` list (string) and a boolean `is_architecture_locked`.
2.  **Enum Updates**: `FlowStatus` in `src/enums.py` will require new states such as `CRITIC_REJECTED` and `ARCHITECTURE_APPROVED`.
3.  **Prompt Template**: The `ARCHITECT_CRITIC_INSTRUCTION.md` will serve as the rigid contract for the critic node. It must explicitly enforce checking for N+1 problems, race conditions, scalability bottlenecks, security risks, and strictly demand that every `SPEC.md` defines unalterable function signatures, class names, and API payloads.
4.  **Immutability**: Once the loop resolves and `is_architecture_locked` becomes true, the generated specification files (`SYSTEM_ARCHITECTURE.md` and the individual `SPEC.md` files) are treated as read-only contracts for the remainder of the project. Any attempt by a later Coder session to modify these contracts will be blocked.

## 4. Implementation Approach
The implementation will follow a precise, step-by-step methodology to integrate the Self-Critic loop into the existing `gen-cycles` command without destabilizing the current CLI or orchestrator.

1.  **Define the Instructions**: First, create the `ARCHITECT_CRITIC_INSTRUCTION.md` template in `src/templates/`. This file must contain the exhaustive checklist of architectural anti-patterns and the strict requirement for interface locking.
2.  **Update Enums**: Modify `src/enums.py` to include `FlowStatus.CRITIC_REJECTED` and `FlowStatus.ARCHITECTURE_APPROVED`.
3.  **Implement the Node Logic**: In `src/graph_nodes.py`, implement the `architect_critic_node` function. This function must retrieve the current state, load the `ARCHITECT_CRITIC_INSTRUCTION.md` template, format it with the current architectural drafts, and send it to the existing Jules session via `jules_client`.
4.  **Implement the Validation Parser**: Within the new node, implement a rigid parser using Pydantic validation (in `src/validators.py`) to interpret Jules's response. The response must be forced into a structured JSON format indicating either 'pass' or 'fail' along with specific, actionable feedback if it fails.
5.  **Modify the Graph**: Finally, update `src/graph.py`. Locate the `_create_architect_graph` method. Change the edge `architect_session -> END` to point to the new `architect_critic_node`. Add conditional edges from the critic node: if `ARCHITECTURE_APPROVED`, route to `END`; if `CRITIC_REJECTED`, route back to `architect_session` with the feedback appended to the state.

## 5. Test Strategy
Testing this cycle requires verifying both the prompt formatting and the state machine routing.

**Unit Testing Approach**: We will write unit tests using `pytest` specifically targeting the `architect_critic_node` function in isolation. We will mock the `JulesClient` to return predefined responses. We must assert that when the mock returns a 'fail' response, the node correctly updates the `CycleState` with `FlowStatus.CRITIC_REJECTED` and appends the feedback. Conversely, we must assert that a 'pass' response sets `FlowStatus.ARCHITECTURE_APPROVED`. We will also unit test the Pydantic parser in `validators.py` to ensure it robustly handles malformed JSON or unexpected responses from the simulated LLM, throwing appropriate exceptions rather than crashing the orchestrator.

**Integration Testing Approach**: We will execute the `gen-cycles` command (via the CLI interface or directly invoking the graph) using a highly constrained, mocked LLM provider designed to fail the first review and pass the second. We will assert that the execution correctly enters the retry loop, that the feedback is correctly injected into the subsequent `architect_session_node` invocation, and that the final output specifications are successfully written to the `dev_documents` directory only after the `ARCHITECTURE_APPROVED` state is reached. We will use temporary directories for this test to ensure no side effects on the actual file system.
