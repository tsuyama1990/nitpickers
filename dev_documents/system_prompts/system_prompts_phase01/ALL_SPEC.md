
# Project Specifications

NTPICKERS 要求定義書 (NEXUS-CDD Architecture Specification)

1. プロジェクト概要と基本方針

本プロジェクトは、既存の自律型開発フレームワーク「AC-CDD」をベースとし、並行開発（Concurrent Development）と厳格な検証（Verifiable Execution）を統合した次世代アーキテクチャ「NITPICKERS」へとアップグレードするものです。
名称の由来である「Nitpicker（粗探しをする人、細部まで徹底的にこだわる人）」の通り、AIが生成したコードの些細な欠陥すら決して許容せず、静的解析・動的テスト・レッドチーム監査を通じて、人間が書くよりも堅牢で完璧なソフトウェアを構築することを目的とします。このシステムは単なるコードジェネレーターではなく、要件定義からデプロイ可能なテスト済みコードの結合までを自律的に行う「仮想開発チーム」として機能します。

1.1 背景と期待される効果

ベースとなる既存のAC-CDDのアーキテクチャは、どれほど強力なモデルを使用しても根本的な構造として「直列的（Sequential）なボトルネック」を抱えていました。一つの機能モジュールの実装とエラー修正が終わるまで、プロジェクト全体の進行が完全に停止してしまいます。また、AIが「目視」でコードをレビューするだけでは、実行時エラー、型定義の不整合、あるいは論理破綻（ハルシネーション）を完全に防ぐことはできず、最終的に人間がデバッグを引き受ける羽目になります。

NITPICKERSは、モダンなDevOpsのベストプラクティスをAIのワークフローに適用することで、以下のブレイクスルーをもたらします。

開発速度の劇的向上 (Massive Throughput): この圧倒的なスピードの根幹は、単なる非同期処理の導入ではなく、上流工程での「完璧な事前定義（Blueprintの作成）」とそれに基づく「コンカレント（並行）開発」の融合にあります。まずプロジェクト全体を複数の機能サイクルに分割し、各サイクルの役割と外部インターフェースを厳格に定義・ロックします。この「揺るぎない契約（Contract）」が事前に存在するからこそ、各AIエージェントは他のサイクルの進捗や他人のコードの完成を待つことなく、自分の担当領域（例えば、認証API、DBスキーマ、UIコンポーネントなど）の実装に100%集中できます。結果として、6つの機能サイクルを非同期で完全に同時に走らせるコンカレント開発が成立し、手戻りや待ち時間なしに理論上最大6倍のスループットを実現します。これは「6人の優秀なエンジニアが、完璧な設計図を元に一切のコミュニケーションコストなしで同時にタイピングしている状態」に匹敵し、従来の直列的な開発フローを根底から覆すスピードを生み出します。

品質の確実な担保 (Zero-Trust Validation): LLMの出力結果を無条件に信用しません。具体的には、LLMがコードを生成した直後に2つの「物理的な関門（ハードゲート）」を設けます。第一の関門として、ruffによる構文・規約チェックとmypyによる厳格な型チェックを自動実行し、エラーがあれば軽量モデル（Fast Model）に即時修正させます。第二の関門として、E2Bサンドボックスという隔離環境内でpytestなどのテストランナーを起動し、事前に定義した受け入れテスト（UAT）を実際に実行します。これらの静的解析と動的テストの双方を完全にクリアした（Greenになった）コードのみを次工程へ進めることで、完遂率（成功率）を100%に近づけます。

アーキテクチャの自己進化 (Evolutionary Refactoring): 並行開発で生じるGitのコンフリクトを「エラー」ではなく「AIによるコード洗練の機会（Abundanceの回収）」と捉えます。これを実現するため、各サイクルの開発の終了後、マージする際のコンフリクト発生時にマージを中断するのではなく、競合マーカー（<<<<<<<など）を含んだコードを専用のAIセッション（マスターインテグレーター）に読み込ませます。AIは双方のブランチの要件と実装意図をコンテキストとして解釈し、「A案とB案の単なる妥協」ではなく、「共通化できる関数はまとめる（DRY原則）」「より効率的なデータ構造を採用する」といった大局的な視点からリファクタリングを行い、マージのたびにコードベースが美しくなる自己組織化設計を実現します。

1.2 【絶対遵守の開発方針（No Zero-Base Rewrite）】

既存のLangGraph構造（graph.py, jules_session_graph.py）、Git操作基盤（git_ops.py）、およびファイル管理基盤は、すでに安定稼働している極めて優秀な資産であるため、破壊的なゼロベースの書き換えを固く禁じます。

理由: 既存のPydanticAIを利用した型安全なプロンプトチェーンや、ローカルファイルシステム・クラウド環境間の同期メカニズムは、数々のトライアンドエラーとエッジケースの解消を経て構築されたものです。これをAIが安易に再構築しようとすると、過去に解決済みのバグが再発し、無用な技術的負債と大幅なプロジェクトの後退を招きます。

原則: 改善は必ず「モジュール指向・アドオン指向」で行うこと。既存ノードの内部ロジックを拡張する（例：プロンプトの注入を追加する）か、新規のルーターやノードを追加してグラフのエッジ（遷移）を繋ぎ変えることのみを許可します。既存の関数のシグネチャを大きく変更する場合は、後方互換性を維持してください。

2. コア機能要件と既存資産の活用マッピング

各要件において、既存コードがどこまでカバーしているか（<実装済み>）を定義し、今回の開発スコープ（【追加実装】）と具体的な振る舞い、エッジケースへの対応策を明確化します。

要件1: サイクル分割計画と自己批判的レビュー (Planning & Self-Critic)

概要: プロジェクトを複数のサイクルに分割し、要件定義の段階でアーキテクチャの妥当性をAI自身にレビュー（レッドチーム化）させます。実装フェーズに入ってから「この設計では要件を満たせない」と気づく手戻りを完全に防ぐための上流工程の強化です。本プロセスは、初期のサイクル計画フェーズ（gen-cycles コマンド実行時）のフローに統合されて実行されます。

現状:

<実装済み>: gen-cycles における Architect エージェントによる全体構造（SYSTEM_ARCHITECTURE.md）と個別機能（SPEC_cycleXX.md）の生成基盤。

<実装済み>: 同フェーズにおける QA Design エージェントによる受け入れテスト（UAT_cycleXX.md）の生成基盤。

【追加実装と詳細仕様】:

Self-Critic Nodeの追加とリスク抽出メカニズム: gen-cycles の実行パイプライン内に、設計を検証するための評価ノードを追加します。指摘された脆弱性は自動的にSPECにフィードバックされ、再生成を通じて設計が強化されます。

実行主体 (同一セッションのJules): 外部の軽量モデルや別エージェントを立ち上げるのではなく、最も推論能力が高く、ここまでの設計コンテキストを完全に保持しているJulesセッションをそのまま活用します。同一セッションを継続するため起動のオーバーヘッドはかかりません。

固定化プロンプトによる徹底的な総点検 (Static Prompts for Broad Verification): 外部AIに依存して質問を生成させるのではなく、ローカル側（Pythonスクリプト）で「あらゆるアンチパターンとエッジケースを網羅した固定化された検証プロンプト」を生成し、それをJulesに直接投げ込みます。Jules自身に「Red Team」として自己の設計を攻撃させます。

コンテキストの結合: 全体構造 (SYSTEM_ARCHITECTURE.md) と対象サイクルの詳細 (SPEC_cycleXX.md) の両方を評価対象とします。

脆弱性チェックリストの強制: ローカルで組み立てたプロンプト内に「DBのN+1問題」「競合状態（Race Condition）」「スケーラビリティのボトルネック」「権限昇格・バリデーション漏れのセキュリティリスク」など、代表的なアンチパターンのリストを明示し、それぞれに対して検証を義務付けます。ARCHITECT_CRITIC_INSTRUCTION.mdを参考にして過不足があれば補正せよ。また、各cycleごとのSPEC.mdの整合性がコンカレント開発では重要になる。したがって整合性、一貫性の検証の徹底をするように念押しをさせること。

推論ステップ (Chain of Thought) の指定: 例えばN+1問題を発見させるため、「① 対象APIがリストや配列を返すか確認する → ② その各要素に関連する別テーブルからのデータ取得が必要か確認する → ③ Eager Loading (JOIN等) の要件が設計書に明記されているか判定する」という段階的な思考プロセスをプロンプトで強制させ、見落としを防ぎます。

Interface Lock（インターフェースの絶対固定）: gen-cycles による各サイクルの SPEC_cycleXX.md 出力時に、「外部に公開する関数名、クラス名、APIのペイロード構造、戻り値の型」を明確に記述させます。並行開発時、各エージェントはこのインターフェース定義を「強い契約（Contract）」として扱い、絶対に勝手に変更してはいけません。これを破った場合、後続のLinterやTDDフェーズで即座に弾かれる仕組みを構築し、並行開発時の結合テストにおける致命的な不整合を防ぎます。

要件2: 全サイクルの並行開発 (Concurrent Development)

概要: 第3プロセス（実装フェーズ）において、全サイクルを直列ではなく並列で同時に実行し、開発速度を最大化します。人間が行うコンカレントエンジニアリングのベストプラクティスをAIエージェントの世界に持ち込みます。

現状:

<実装済み>: workflow.py におけるサイクル実行のループ構造（現在はSequentialに1つずつ実行し、完了を待つ仕様）。

<実装済み>: GitManager によるフィーチャーブランチ（feature/cycle-01など）の独立した作成・切り替え機能。

【追加実装と詳細仕様】:

非同期ディスパッチャーの実装: workflow.py の同期的な for ループを廃止し、asyncio.gather や並行実行用のタスクキューを用いて、複数のJulesセッション（クラウドコンピュート）を同時に呼び出します。これにより、I/O待ちや推論待ちの時間が完全にオーバーラップします。なお、各サイルの実装フローにおいて同一のAUDITORが2回ずつ監査を実施しますが、これらは必ず「同じセッション（同一コンテキスト）」を維持して呼び出すよう強く念押しし、監査の基準や文脈が途中でブレないように完全な同一人物として機能させます。

DAG（有向非巡回グラフ）スケジューラー: オプションとして、単なる「よーいドン」の全並列ではなく「依存関係」を考慮したスケジューリングを行います。例えば「サイクル1（DBスキーマ設計）」がマージされなければ「サイクル2（APIエンドポイント実装）」は開始できないように、トポロジカルソートに基づいてJulesをキックする賢いディスパッチャーを導入し、無意味な実装の空振りを防ぎます。

APIレート制限対策とバックオフ: 多数のLLMリクエストが同時に飛ぶため、OpenRouterやJules APIから HTTP 429 (Too Many Requests) エラーが返るリスクが高まります。これを捕捉し、ジッター（揺らぎ）付きのバックオフ（待機時間）で安全にリトライするセーフティネットをネットワーク層に組み込みます。

要件3: サイロ内開発のレッドチーム監査 (Red Teaming Intra-cycle)

概要: 各サイクルの並行開発中、Auditorが厳格なプロンプトを用いて自サイクルのコードをレビューし、アーキテクチャの劣化やコーディング規約違反を防ぎます。

現状:

<実装済み>: auditor_node による静的コードレビューと、指摘事項に基づく修正ループ（3名の監査人が各2回ずつ、合計最大6回のJules修正ループを実施）。

<実装済み>: LangGraph及びPydantic を用いた FAST_MODEL (OpenRouter等) の呼び出し基盤 (agents.py)。

【追加実装と詳細仕様】:

監査前のリファクタリングおよび、Self-Critic Nodeの追加: Auditorの監査前に固定プロンプトを送付することで実装後リファクタリングの余地があるかの確認おさせる (POST_AUDIT_REFACTOR_INSTRUCTION.mdを参考に適切に直せ)。その後設計、スキーマやコードを検証するための評価ノードを追加します (CODER_CRITIC_INSTRUCTION.md)。

要件4: サンドボックスでのUAT検証 (Sandbox UAT Verification)

概要: AIの「自分のコードは完璧だ」というハルシネーション（ブラックボックス）を排除するため、独立したE2B環境内で実際にテストスクリプトを実行し、客観的で検証可能な証拠（Execution Artifacts）をプロジェクトの進行要件とします。

現状:

<実装済み>: E2B環境のコンテナセットアップ基盤と、グラフ内のプレースホルダーノード (uat_evaluate_node)。

<実装済み>: QA Designフェーズで生成された、実行すべきテストシナリオ (UAT_cycleXX.md) の存在。

【追加実装と詳細仕様】:

動的実行パイプラインの構築 (ローカルとサンドボックスの連携): uat_evaluate_node を本格稼働させ、ローカルのオーケストレーターとE2Bをシームレスに連携させます。

ワークスペース同期: ローカル環境（LangGraphの実行ホスト）からE2B SDKを利用し、AIが生成した実装ソースコードとテストコードをサンドボックス内の仮想ファイルシステムに同期（マウントまたはプッシュ）します。

テストコードの生成と実行: 自然言語で書かれた UAT_cycleXX.md をインプットとし、AIに pytest 準拠のテストスクリプト（test_*.py）を生成させます。その後、ローカル側からE2Bに対して pytest -v --tb=short などの実行コマンドを発行し、テストプロセスをキックします。

アーティファクトの抽出: E2Bの実行コンテナから、標準出力（stdout）、エラー出力（stderr）、終了コード（Exit Code）、およびコードカバレッジレポートをローカル側で吸い上げ、プロンプトのコンテキストとして利用できるようLangGraphの CycleState に保存します。

Agentic TDD (テスト駆動開発) の徹底: Julesが本番コードを書く「前」に、必ずテストが『失敗（Red）』することを確認し、実装後に『成功（Green）』することをもってタスク完了とする厳格な「Red-Green-Refactor」サイクルをAIに回させます。

Step 1 (Redの保証): 実装コードのモック（passやNotImplementedErrorのみを持つ関数）とテストコードのみを先に生成させ、E2B内で実行します。テストが確実に「失敗」することを確認します。もし成功してしまった場合は「テストのAssertionが機能していない・甘すぎる」として、テストスクリプト自体を再生成させます。

Step 2 (Greenへの到達と自己デバッグ): テストの妥当性が確認できたら、本番のロジックを実装させ、再度E2Bでテストを実行します。失敗した場合、ローカルのノードがE2Bから得られた生のエラートレースバック（スタックトレース）を抽出し、「テスト結果: 失敗。以下のトレースバックを解析して実装を修正せよ」というプロンプトを動的に組み立て、要件3の修正ループ（Jules）へ差し戻します。これにより、AIは想像ではなく物理的な証拠に基づいて「なぜ失敗したのか」を自己デバッグできるようになります。

要件5: コンフリクトのカスケード解消 (Cascading Merge Resolutions)

概要: 並行開発されたフィーチャーブランチを統合（マージ）する際、Gitの自動マージ（テキストの行ベースの合成）に頼らず、AI（Jules）の高い文脈理解力を生かして「意味的なコンフリクト解消（Semantic Merge）」を行います。

現状:

<実装済み>: サイクル完了時の単純な git merge 実行ロジック（コンフリクト発生時に停止するリスクあり）。

【追加実装と詳細仕様】:

Git-Native Conflict Extraction: コンフリクト発生時、マージをAbort（中止）するのではなく、<<<<<<< HEAD や ======= といったGit標準の競合マーカーが含まれたファイルをそのまま維持します。Python側のスクリプトで競合が発生しているファイルをリストアップし、関連する双方のブランチのSPEC情報と合わせてJulesが読みやすいコンテキストパッケージを作成します。

Conflict Registry (追跡管理と検証機構): コンフリクトの解消漏れ（ハルシネーションによるマーカーの放置や修正漏れ）を物理的に防ぐため、ローカル側（オーケストレーター）でコンフリクト発生ファイルとその状態をBookkeeping（帳簿管理）します。Julesから修正案が返却された際、ローカルのパーサーが該当ファイル内に競合マーカー（<<<<<<<, =======, >>>>>>>）が残っていないか正規表現等で厳格にチェックします。もし残留があれば「未解消のマーカーが残っています」としてJulesに即時差し戻すフェールセーフを設けます。

Stateful Integrator Session (長寿命統合セッション): マージのたびに新しいセッションを立ち上げるとAPI制限やコンテキストの喪失が発生します。そのため、統合専用のJulesセッションを「マスターインテグレーター」として1つだけ立ち上げます。Python側から順番にコンフリクトのパッケージをこのセッションに投げ込み、「単なるテキストの合体ではなく、双方の意図を汲み取り、より優れたアーキテクチャになるようマーカーを消して統合せよ」と指示します。このセッションは過去のマージの記憶を保持しているため、プロジェクト全体の整合性が保たれます。

要件6: 全体コードのリファクタリングと最適化 (Overall Refactor)

概要: 全てのサイクルのマージが完了した後、個別の並行開発（サイロ化された作業）で生じた「局所最適化」を俯瞰し、システム全体の「全体最適化（Abundanceの回収）」を行います。

【追加実装と詳細仕様】:

Global Refactor ノードの追加: 要件5の最終ステップ（すべてのブランチが main または integration にマージされた直後）として実行されます。全ソースコードのAST構造と SYSTEM_ARCHITECTURE.md をインプットとし、プロジェクト全体を大局的な視点で監査します。

DRY原則（Don't Repeat Yourself）の適用: 例えば、Cycle 01とCycle 03がそれぞれ独自の「日付フォーマット変換関数」や「APIレスポンスのラップ関数」を作ってしまった場合、それらを検知し、一つの utils/helpers.py 等に統合します。また、不要になった循環参照やデッドコードをパージし、最終的な本番リリースに耐えうるクリーンなコードベースに仕上げる高度なリファクタリングを自律的に行わせます。

要件3のフローに従う: リファクタリング実施後、要件3のフローに従い、Self-critic Review, Audit (2回 x 3名の監査人), Self-Critic reviewの順番でコードの死角を徹底的にたたく。
