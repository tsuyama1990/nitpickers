# CYCLE02 Specification

## Summary
The primary objective of CYCLE02 is to implement Strategy 1 of Phase 1: The Inner Loop (Structural Integrity & TDD), specifically focusing on establishing "Docs-as-Tests" and Pytest Orchestration. This cycle addresses a critical vulnerability in agent-driven development: the "Translation Gap," where high-level markdown specifications are often misinterpreted, hallucinated, or incompletely implemented by LLMs during the transition to executable test code. By implementing a robust pre-processing mechanism to natively extract and temporarily instantiate specific `uat-scenario` code blocks directly from the `ALL_SPEC.md` or `README.md` documents, we enforce a direct, mechanical link between the user's documented requirements and the system's active test suite. This integration ensures that the generated application code is continuously validated against the exact, unmodified source of truth provided by the user, thereby establishing a rigorous, uncompromising Test-Driven Development (TDD) foundation. This cycle guarantees that documentation is no longer just passive text, but an active, executable artifact, fundamentally strengthening the structural integrity of the resulting codebase and paving the way for the dynamic execution phase in subsequent cycles. By treating documentation as a primary source of executable logic and employing temporary file generation, we drastically reduce the cognitive load on the LLM agents and ensure deterministic compliance with the original human intent while avoiding the complexities of custom Pytest Item manipulation.

## System Architecture
The system architecture for this cycle is deeply integrated into the Pytest execution lifecycle, but pivots from a complex dynamic hook to a robust, pre-execution file generation strategy. To avoid the known brittleness and isolation issues associated with yielding custom Pytest `Item` instances with manipulated namespaces, we will introduce a pre-processing step (which could be invoked via a simpler Pytest fixture or a wrapper script called by `ProcessRunner`). This mechanism will scan the project's root directory for designated markdown files, primarily targeting `ALL_SPEC.md`. A custom parsing utility will extract pure Python code blocks explicitly annotated with a `uat-scenario` tag. The crucial architectural shift is that these extracted blocks will be dynamically written to temporary, standard `.py` test files within a designated `tests/generated_docs/` directory immediately prior to the primary Pytest invocation. This allows Pytest to discover, collect, and execute these files using its completely standard, native mechanisms. The architectural boundary here is explicitly clear: the pre-processor acts as a secure, deterministic compiler, transforming static markdown into standard Python test modules. It ensures that the existing, robust test runner infrastructure is fully utilized without custom monkey-patching. This integration provides a mechanical, verifiable guarantee that the implementation aligns perfectly with the documented specifications, replacing assumed success with deterministic verification. This robust architectural integration guarantees that the Docs-as-Tests paradigm seamlessly coexists with traditional unit tests, providing standard tracebacks and reporting out of the box.

## Design Architecture
The design architecture for CYCLE02 focuses on defining the internal, strongly-typed data structures required to parse and represent markdown-based tests reliably before they are compiled into temporary Python files. We will introduce a lightweight, precise Pydantic schema, `MarkdownTestBlock`, to strictly define the structure of the extracted scenarios. This schema will enforce critical constraints, such as the required presence of an execution language tag (ensuring it is `python`), a unique, parsed scenario identifier (used to generate deterministic temporary filenames like `test_scenario_1.py`), and the raw, executable code payload itself. By utilizing `ConfigDict(extra='forbid')`, we guarantee that only valid, well-formed markdown blocks are processed by the file generator. This strict validation prevents malformed documentation, syntax errors in the markdown, or unrecognized tags from causing unhandled exceptions during the pre-processing phase. The primary consumer of this schema will be the custom parser function located within the new `docs_preprocessor.py` utility. This design approach maintains perfect backward compatibility with the existing Pytest configuration and standard unit tests while safely extending its capabilities to include documentation. It adheres firmly to the project's foundational Schema-First philosophy, ensuring the system's stability is never compromised by input variability and that the generated temporary files are structurally sound before Pytest touches them.

## Implementation Approach
The implementation plan for CYCLE02 begins with the precise definition of the `MarkdownTestBlock` Pydantic schema to strictly type and validate the code blocks extracted from the specification documents. Following the schema definition and its unit tests, the core development effort will center on creating the `tests/docs_preprocessor.py` module. We will implement logic utilizing regular expressions or a dedicated markdown parsing library to identify fenced code blocks tagged for UAT execution in `ALL_SPEC.md`. These blocks will be extracted, validated against the `MarkdownTestBlock` schema. Crucially, the implementation will then iterate through the validated blocks and write them to physical, temporary files within a `tests/generated/` directory (ensuring this directory is added to `.gitignore`). We will then update `tests/conftest.py` or the `ProcessRunner` command invocation to ensure this pre-processor runs automatically before Pytest test collection begins. Pytest will then natively discover and run these generated `test_*.py` files. Finally, we will implement cleanup logic (e.g., via a Pytest teardown hook) to automatically delete the `tests/generated/` directory after the test suite completes, ensuring a clean workspace. Thorough error handling must be implemented within the parser to gracefully ignore malformed blocks while clearly logging warnings.

## Test Strategy

### Unit Testing Approach
The unit testing strategy for this cycle will rigorously validate the intricate markdown parsing logic and temporary file generation mechanisms in complete isolation. We will write explicit, comprehensive unit tests for the custom parser function located within `docs_preprocessor.py`. These tests will provide the parser with simulated, hardcoded markdown strings containing various permutations: valid scenarios, invalid syntax, missing tags, and malformed code blocks. We will assert that the parser correctly extracts and validates the code payloads against the `MarkdownTestBlock` schema under all conditions. We will extensively use `unittest.mock.patch` to simulate file reading and writing operations (specifically mocking `builtins.open`), preventing tests from depending on actual documentation or writing real files during the unit test phase. Additionally, we will verify that the generated Python source code strings are formatted correctly and maintain proper indentation. All mocked interactions will be rigorously type-hinted to pass the project's strict `mypy` requirements, ensuring the pre-processor is fully resilient and type-safe. This ensures that the foundational components of the Docs-as-Tests feature are thoroughly validated before they interact with the live file system.

### Integration Testing Approach
The integration testing strategy for CYCLE02 will simulate a complete, end-to-end Pytest run to verify that the file-generation Docs-as-Tests integration functions flawlessly within the broader testing ecosystem. We will create a dedicated, temporary project structure containing a mock `ALL_SPEC.md` file populated with predefined, executable UAT scenarios. We will then programmatically invoke the pre-processor and the Pytest runner on this temporary directory. We will assert that the temporary `test_*.py` files are successfully created on disk, that Pytest natively discovers and executes them alongside standard tests, and that standard Pytest reporting (stdout) accurately reflects the pass/fail status of these specific markdown tests. Furthermore, we will simulate a critical scenario where a markdown test deliberately fails due to an assertion error. We will ensure that the resulting traceback correctly points to the lines within the generated temporary file, which natively maps back to the markdown logic. Finally, the test must verify that the temporary files are successfully deleted upon test completion, leaving the directory clean. This end-to-end integration test is paramount for confirming that the structural integrity of the TDD pipeline is mechanically enforced directly from the requirement specifications.
