# CYCLE02 SPECIFICATION

## Summary
This cycle implements Phase 0: The Environment and Observability Setup Gate. Real end-to-end tests require local secrets and hardware contexts. Tracing must be active from the very first execution to ensure complete observability of the UAT pipeline. We will introduce a mechanical blockade in the CLI that prevents the execution of the agent loop unless the necessary LangSmith environment variables and API keys are verified. This ensures a true zero-trust validation environment where no blind execution is permitted. The primary goal is to parse and validate the `.env` configuration robustly, mapping it to the Pydantic-based settings schema before instantiating the LangGraph orchestration layer. The environment gatekeeper acts as the foundational layer for all subsequent phases, guaranteeing that when the dynamic Playwright UATs fail, the Observability Layer (LangSmith) successfully tracks the node transitions and raw Vision LLM prompts generated by the Stateless Auditor.

## System Architecture
The architecture introduces an interception point within `src/cli.py` or the primary `ManagerUseCase`. Before triggering the `run-cycle` phase (the execution of the Stateful Worker), the system performs an automated dependency scan. It scans the `SPEC.md` documents for required external dependencies (e.g., `DATABASE_URL`, `OPENAI_API_KEY`) and validates their presence in the local environment. Furthermore, it strictly enforces the LangSmith setup by requiring `LANGCHAIN_TRACING_V2=true`, `LANGCHAIN_API_KEY`, and `LANGCHAIN_PROJECT` to be correctly populated.

If any validation fails, a hard stop prompt is presented to the user, immediately halting the CLI command. This prevents the workflow from advancing to the inner or outer loops without complete observability instrumentation. The design explicitly avoids God Classes by ensuring this validation logic is encapsulated within a dedicated `EnvironmentValidator` or extended `BaseSettings` object, adhering strictly to the separation of concerns. The CLI solely invokes the validation and handles the user-facing output, while the domain models dictate the required fields.

```text
/
├── src/
│   ├── **cli.py**
│   └── services/
│       └── **workflow.py**
├── src/domain_models/
│   └── **config.py**
└── **.env.example**
```

## Design Architecture
The design relies on Pydantic's `BaseSettings` for robust environment variable validation. We will extend the configuration models, likely located in `src/domain_models/config.py` or equivalent, to explicitly require the LangSmith observability keys. The environment gatekeeper logic acts as a strict validator, catching misconfigurations at the boundary before they propagate into the agent execution context.

This model aligns with the zero-trust principle, ensuring the application cannot run in a degraded observability state. The Pydantic model will define `langchain_tracing_v2: bool = Field(default=False)` (which must be true for execution), `langchain_api_key: str = Field(min_length=1)`, and `langchain_project: str = Field(min_length=1)`. The CLI command will instantiate this settings object inside a `try/except ValidationError` block, transforming raw Pydantic errors into the specific, user-friendly Hard Stop Prompt required by the specification.

## Implementation Approach
The implementation approach involves updating `.env.example` with the new LangSmith templates (`LANGCHAIN_TRACING_V2=true`, `LANGCHAIN_API_KEY=...`, `LANGCHAIN_PROJECT=nitpickers-uat-pipeline`).

**Step 1:** Modify the configuration model in `src/domain_models/config.py` or where environment settings are parsed. Inherit from `BaseSettings` and add the three required LangSmith variables with strict validation constraints. Add an alias generator if necessary to map the uppercase environment keys to lowercase Python properties.

**Step 2:** In `src/cli.py` or `src/services/workflow.py`, inject the Phase 0 Environment & Observability Gate logic into the command that triggers the worker execution (e.g., `run_cycle()`). We will instantiate the updated settings model.

**Step 3:** If a `ValidationError` is caught during instantiation, catch it explicitly. Use Typer's error styling to output the exact Hard Stop Prompt: `⚙️ Cycle planning complete. Please ensure required secrets (e.g., API keys) AND your LangSmith tracing variables (LANGCHAIN_TRACING_V2=true, LANGCHAIN_API_KEY, LANGCHAIN_PROJECT='nitpickers-uat-pipeline') are correctly populated in your local '.env' file before proceeding with the run-cycle phase.` Then, execute a hard `sys.exit(1)` to abort the workflow mechanically.

## Test Strategy

### Unit Testing Approach
The unit tests will verify that the settings model correctly raises a `ValidationError` when the LangSmith variables are absent or incorrectly formatted. We will use `unittest.mock.patch.dict(os.environ, ...)` to simulate various environment variable states (missing keys, empty string values, correct configuration) and assert the configuration parser behaves deterministically without directly mutating the global `os.environ` during test suite execution. We will also write specific tests verifying that the exact Hard Stop Prompt string is generated by the CLI handler when the validation error is caught.

### Integration Testing Approach
The integration tests will invoke the CLI command simulating the `run-cycle` phase using Typer's `CliRunner`. We will execute the command against a mock `.env` context. We will capture standard output and standard error to verify that the application halts with a non-zero exit code and prints the exact Hard Stop Prompt when the environment is improperly configured. Conversely, we will verify that when the mock context includes all required tracing variables, the CLI successfully bypasses the gate and invokes the workflow service, proving the interception point functions as an impenetrable but passable gate.
