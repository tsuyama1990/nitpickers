# CYCLE02 Specification

## Summary
The primary objective of CYCLE02 is to implement Strategy 1 of Phase 1: The Inner Loop (Structural Integrity & TDD), specifically focusing on establishing "Docs-as-Tests" and Pytest Orchestration. This cycle addresses a critical vulnerability in agent-driven development: the "Translation Gap," where high-level markdown specifications are often misinterpreted, hallucinated, or incompletely implemented by LLMs during the transition to executable test code. By implementing a robust mechanism to natively parse and execute specific `uat-scenario` code blocks directly from the `ALL_SPEC.md` or `README.md` documents, we enforce a direct, mechanical link between the user's documented requirements and the system's active test suite. This integration ensures that the generated application code is continuously validated against the exact, unmodified source of truth provided by the user, thereby establishing a rigorous, uncompromising Test-Driven Development (TDD) foundation for the entire automated UAT pipeline. This cycle guarantees that documentation is no longer just passive text, but an active, executable artifact, fundamentally strengthening the structural integrity of the resulting codebase and paving the way for the dynamic execution phase in subsequent cycles. By treating documentation as a primary source of executable logic, we drastically reduce the cognitive load on the LLM agents and ensure deterministic compliance with the original human intent.

## System Architecture
The system architecture for this cycle is deeply integrated into the existing Pytest framework, introducing custom, sophisticated hooks to fundamentally alter the test discovery and execution process without requiring a separate, proprietary test runner. Specifically, we will modify the `tests/conftest.py` file to implement the `pytest_collect_file` hook. This crucial hook will be designed to scan the project's root directory during the test collection phase for designated markdown files, primarily targeting `ALL_SPEC.md` and potentially `README.md`. Upon locating these specific files, a custom parsing utility will be invoked to extract specific code blocks that are explicitly annotated with a `uat-scenario` tag or a similar designated marker. These extracted blocks, containing pure Python code, will then be dynamically yielded back to Pytest as native, executable Pytest `Item` instances. This architectural approach elegantly bypasses the need for an intermediary, error-prone LLM translation step, allowing Pytest to directly consume the documentation as a primary source of executable tests. The architectural boundary here is explicitly clear: the `conftest.py` acts as the secure bridge between static markdown documentation and dynamic, sandboxed execution. It ensures that the existing, robust test runner infrastructure is fully utilized. This integration will significantly enhance the structural integrity of the pipeline, providing a mechanical, verifiable guarantee that the implementation aligns perfectly with the documented specifications, a critical component of the overarching goal to replace assumed success with deterministic, automated verification. This robust architectural integration guarantees that the Docs-as-Tests paradigm seamlessly coexists with traditional Python unit tests within a unified reporting framework.

## Design Architecture
The design architecture for CYCLE02 focuses on defining the internal, strongly-typed data structures required to parse, represent, and execute markdown-based tests reliably within the Pytest ecosystem. While Pytest inherently uses its own complex object model for item representation, we will introduce a lightweight, precise Pydantic schema, `MarkdownTestBlock`, to strictly define the structure of the extracted scenarios before they are yielded to the Pytest engine. This schema will enforce critical constraints, such as the required presence of an execution language tag (e.g., ensuring it is `python`), a unique, parsed scenario identifier for clear reporting, and the raw, executable code payload itself. By utilizing `ConfigDict(extra='forbid')`, we guarantee that only valid, well-formed, and expected markdown blocks are passed to the Pytest execution engine. This strict validation prevents malformed documentation, syntax errors in the markdown, or unrecognized tags from causing unhandled exceptions or obscure failures during the test collection phase. The primary consumer of this schema will be the custom parser function located within `conftest.py`. This design approach maintains perfect backward compatibility with the existing Pytest configuration and standard unit tests while safely extending its capabilities to include documentation as a first-class, verifiable testing artifact, adhering firmly to the project's foundational Schema-First philosophy and ensuring the system's stability is never compromised by input variability. This rigorous typing ensures that the bridge between static text and dynamic execution is safe, predictable, and strongly validated before execution begins.

## Implementation Approach
The implementation plan for CYCLE02 begins with the precise definition of the `MarkdownTestBlock` Pydantic schema to strictly type and validate the code blocks extracted from the specification documents. Following the schema definition and its corresponding unit tests, the core development effort will center on updating the `tests/conftest.py` file. We will implement the `pytest_collect_file` hook, instructing it to identify and parse target markdown files like `ALL_SPEC.md`. Within this hook, a custom parser utilizing regular expressions or a dedicated markdown parsing library will identify fenced code blocks explicitly tagged for UAT execution. These blocks will be extracted, validated against the `MarkdownTestBlock` schema, and then dynamically wrapped into custom Pytest `Item` instances. The next crucial step is defining exactly how these custom items are executed; we will implement the `runtest` method for these items, ensuring they are executed within a safe, isolated context. This may involve leveraging Python's built-in `exec` function within a tightly controlled, empty namespace to prevent scope leakage between tests. Thorough, robust error handling must be implemented within this execution context to ensure that syntax errors or assertion failures within the markdown blocks are captured and reported clearly as standard Pytest failures, complete with tracebacks, rather than opaque collection errors. Finally, we will review and update the project's `pytest.ini` or `pyproject.toml` configurations to ensure the new custom collection logic is properly recognized, prioritized, and executes alongside the standard test suite without conflict. This careful, step-by-step approach ensures maximum compatibility and stability within the testing environment.

## Test Strategy

### Unit Testing Approach
The unit testing strategy for this cycle will rigorously validate the intricate markdown parsing logic and Pytest integration mechanisms in complete isolation. We will write explicit, comprehensive unit tests for the custom parser function located within `conftest.py`. These tests will provide the parser with simulated, hardcoded markdown strings containing various permutations: valid scenarios, invalid syntax, missing tags, and malformed code blocks. We will assert that the parser correctly extracts and validates the code payloads against the `MarkdownTestBlock` schema under all conditions. We will extensively use `unittest.mock.patch` to simulate file reading operations, preventing tests from depending on the actual, potentially changing project documentation files. Additionally, we will construct focused unit tests for the custom Pytest `Item` execution logic. We will verify that the dynamic execution context is correctly established, that namespace isolation is maintained, and crucially, that exceptions raised within the simulated markdown code are appropriately captured and reported as test failures rather than system crashes. All mocked interactions with the Pytest API will be rigorously type-hinted to pass the project's strict `mypy` requirements, ensuring the collection hook is fully resilient and type-safe. This ensures that the foundational components of the Docs-as-Tests feature are thoroughly validated before they interact with the live file system or the active Pytest runner.

### Integration Testing Approach
The integration testing strategy for CYCLE02 will simulate a complete, end-to-end Pytest run to verify that the Docs-as-Tests integration functions flawlessly within the broader testing ecosystem. We will create a dedicated, temporary project structure containing a mock `ALL_SPEC.md` file populated with predefined, executable UAT scenarios. We will then programmatically invoke the Pytest runner on this temporary directory and assert that the custom `pytest_collect_file` hook successfully discovers the markdown file, accurately extracts the scenarios, and executes them as native test items alongside any standard Python test files. The tests will verify that standard Pytest reporting (stdout) accurately reflects the pass/fail status of these specific markdown tests. Furthermore, we will simulate a critical scenario where a markdown test deliberately fails due to an assertion error. We will ensure that the resulting traceback and error logs are correctly captured and formatted by Pytest's standard mechanisms, ready to be piped back to the Worker or Auditor agents in future cycles. This end-to-end integration test is paramount for confirming that the structural integrity of the TDD pipeline is mechanically enforced directly from the requirement specifications. It acts as the definitive proof that the translation gap has been successfully closed.
