# CYCLE04 Specification

## Summary
The primary objective of CYCLE04 is to implement Strategy 3 of Phase 2: The Outer Loop (Behavioral Reality Sandbox), specifically focusing on Playwright Multi-Modal Capture and structured JSON handoffs. This cycle represents a pivotal transition in the UAT pipeline, moving from purely structural verification to dynamic behavioral validation. By integrating `pytest-playwright`, the system executes end-to-end user journeys within a headless browser. Crucially, when these UI tests fail, the system automatically triggers a sequence to capture multi-modal artifacts—high-resolution screenshots and DOM traces. To resolve the architectural weakness of brittle directory scanning, this cycle implements a strict requirement: the Pytest execution hook must not only save the images but must actively generate a `TestReportManifest.json`. This manifest provides an explicit, strongly-typed mapping between the specific Pytest node ID that failed and the absolute paths of its corresponding visual artifacts. This structured handoff serves as definitive proof of failure, providing the Stateless Auditor with exact context and eliminating race conditions in concurrent execution environments.

## System Architecture
The system architecture for this cycle extends the Pytest execution environment through the integration of the `pytest-playwright` plugin and custom reporting hooks. The primary modifications will occur within `tests/conftest.py`, where the `pytest_runtest_makereport` hook dynamically intercepts test failures. The architecture dictates that the Playwright browser context must be managed deterministically, ensuring isolated execution for each test scenario. When a failure is detected, the system will instruct the active Playwright `page` object to capture a screenshot and extract the DOM state, saving these to a standardized `dev_documents/artifacts/` directory. The critical architectural enhancement is the introduction of the `artifact_manifest.json` generation logic within this same hook. Instead of relying on downstream modules to guess which screenshot belongs to which test via timestamps, the hook will deterministically construct a JSON registry linking the `nodeid` to the generated file paths. This centralized artifact manifest serves as the formal integration boundary between the Sandbox Execution phase and the subsequent Orchestrator parsing phase. The implementation must guarantee that the artifact and manifest generation process is highly robust; if a screenshot fails due to a browser crash, the manifest must still be written (perhaps with null paths) to accurately reflect the test failure state, ensuring the pipeline never hangs indefinitely.

## Design Architecture
The design architecture relies on formally defining the concept of a multi-modal test failure and the JSON handoff manifest. We will introduce a rigorous Pydantic schema, `TestReportManifest`, which contains a mapping of test identifiers to `MultiModalArtifact` objects. The `MultiModalArtifact` schema will encapsulate critical information such as the absolute file paths to the locally generated screenshot (`screenshot_path`) and the DOM trace (`trace_path`). By defining this schema with `ConfigDict(extra='forbid')`, we enforce a strict contract: any module downstream (specifically, `uat_usecase.py` in CYCLE05) that consumes this JSON must receive perfectly structured, predictable data. The schema will include active validation logic to verify that the file paths actually exist on the local file system before the object is instantiated. This prevents "file not found" errors during the Auditor's analysis phase. This schema-first design, specifically moving from implicit file dropping to explicit JSON manifesting, ensures that the transition from a broken UI test to the structured input required by an LLM is handled deterministically, safely, and is immune to race conditions caused by parallel test execution.

## Implementation Approach
The implementation plan for CYCLE04 begins with updating `pyproject.toml` to explicitly include `pytest-playwright`. Following dependency installation, we will define the `TestReportManifest` and `MultiModalArtifact` Pydantic schemas to strictly type the JSON output of a failed UI test. The core development effort then shifts exclusively to `tests/conftest.py`. We will implement the `pytest_runtest_makereport` hook to intercept the test execution lifecycle. If a test item utilizing a Playwright `page` fixture fails, the hook will securely trigger the artifact capture sequence, saving the files to `artifacts/`. Crucially, the hook will then instantiate or update a global dictionary mapping the test's `nodeid` to the newly created file paths. At the conclusion of the entire Pytest session (e.g., using the `pytest_sessionfinish` hook), this dictionary will be formally serialized and written to disk as `artifact_manifest.json` within the root or artifacts directory. The final step involves writing targeted unit and integration tests to verify that the hook correctly identifies failures, captures images without throwing secondary exceptions, and successfully generates a strictly valid JSON manifest that complies perfectly with the Pydantic schema, regardless of test execution order.

## Test Strategy

### Unit Testing Approach
The unit testing strategy focuses on isolating the custom Pytest hook and JSON manifest generation logic within `conftest.py`. We will utilize `unittest.mock` to simulate various Pytest execution states, specifically mocking the `pytest_runtest_makereport` and `pytest_sessionfinish` lifecycle events. Tests will be constructed to simulate explicitly failing Playwright tests. For simulated failures, we will inject a mocked Playwright `page` object and verify that the hook correctly calls `page.screenshot()`. We will forcefully assert that the hook correctly populates the internal tracking dictionary and that the session finish hook successfully writes a structurally valid JSON string to the mocked file system. Furthermore, we will rigorously test the `TestReportManifest` Pydantic schema in isolation, ensuring it correctly validates valid JSON strings and file paths, and raises appropriate exceptions for malformed manifests. All mock interactions will be strictly typed to comply with `mypy`, ensuring the artifact capture logic is resilient and type-safe before broader integration.

### Integration Testing Approach
The integration testing strategy involves executing a real Playwright test suite to verify the end-to-end artifact and manifest capture process within a live environment. We will create a temporary test file containing a deliberate UI failure (e.g., asserting visibility of a non-existent element). We will invoke Pytest programmatically. The primary assertion will be verifying that upon test completion, the specified `artifacts/` directory contains a valid `.png` screenshot, and crucially, that an `artifact_manifest.json` file is present in the expected location. The test will read this JSON file, parse it using the `TestReportManifest` schema, and assert that the `nodeid` matches the failed test and that the paths point to the newly generated artifacts. We will further assert that the Pytest execution summary correctly reflects the failure and that the custom hook did not introduce any secondary errors. This integration test provides absolute empirical confirmation of the structured JSON handoff mechanism.
